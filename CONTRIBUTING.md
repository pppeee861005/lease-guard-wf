# 貢獻指南 | Contributing Guide

感謝你想為租約哨兵做出貢獻！本指南說明如何參與這個專案。

---

## 📋 行為準則

我們致力於提供一個安全、尊重的協作環境。任何包含騷擾、歧視或不尊重的言論和行為都不被接受。

## 🔍 回報問題

### 發現 Bug
1. 檢查 [Issues](../../issues) 是否已有相同報告
2. 如果沒有，新建 Issue，包含：
   - 清楚的標題
   - 詳細的複現步驟
   - 預期行為 vs. 實際行為
   - 截圖或合約樣本（已去個資化）
   - 運行環境（OS、Python 版本等）

### 提出功能需求
1. 討論題目：描述需求的場景和價值
2. 等待核心團隊反饋
3. 被接受後開始實現

## 🛠️ 開發流程

### 準備開發環境
```bash
git clone https://github.com/[USERNAME]/lease-guard-wf.git
cd lease-guard-wf
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt -r requirements-dev.txt
```

### 開發步驟
1. 建立新分支：`git checkout -b feature/[描述性名稱]`
2. 實現功能並編寫測試
3. 確保通過所有測試：`pytest`
4. 提交代碼：`git commit -m "feat: 簡短描述"`
5. 推送分支：`git push origin feature/[名稱]`
6. 開啟 Pull Request

### Commit 訊息約定
使用 Conventional Commits 格式：
- `feat:` 新功能
- `fix:` 修復 Bug
- `docs:` 文檔變更
- `style:` 代碼風格（不影響邏輯）
- `refactor:` 代碼重構
- `test:` 測試相關
- `chore:` 構建工具或依賴更新

**例**：
```
feat: 優化 OCR 處理，支援手機斜拍照片

- 添加影像自動校正
- 提升低光照環境識別率（+8%）
```

## 📚 移植到其他國家/地區

「規則庫比對型 Workflow」是通用設計。如果你想在自己國家/地區適配：

1. **Fork 本 repo**
2. **替換規則庫**（`rules/` 目錄）
3. **更新 Stage 5 轉介資源**（本地化組織名單）
4. **修改 OCR 語言設定**
5. **保留三大鐵律原文**（重要！）
6. **補齊本地化的測試集與評測報告**
7. **開啟 Discussion 告訴我們你的計畫**（方便後續社群協作）

詳見：`docs/fork-guide.md`

## 🏭 規則庫維護

### 規則更新工作流（快照+哨兵模式）
1. **哨兵 Agent 每月檢測官方規範更新**
2. **檢測到變動 → 自動開 GitHub Issue**
3. **核心團隊人工審核變動**
4. **變動合併 → 規則庫版本號 +1**
5. **記錄於機房報告**

### 貢獻規則修正
如果你發現規則庫中的錯誤或遺漏：
1. 開啟 Issue，附上：
   - 官方文件連結和截圖
   - 現行規則中的問題
   - 建議修正
2. 等待核心團隊驗證
3. 合併後自動包含在下一版本評測中

## ✅ 代碼審查標準

Pull Request 需符合以下條件才能合併：
- [ ] 代碼通過所有測試
- [ ] 新增功能有對應測試
- [ ] 符合專案風格指南
- [ ] 文檔已更新
- [ ] Commit 訊息清晰
- [ ] 不引入新的依賴（除非必需且已討論）

## 📊 評測貢獻

租約哨兵最需要的幫助是**更多真實合約的評測**：

1. **蒐集合約**：願意分享（去個資化）的合約樣本
2. **人工標注**：按 `eval/annotation-standard.md` 標注風險
3. **結果驗證**：跑系統評測後驗證輸出的準確性
4. **回報漏報**：告訴我們系統漏掉了什麼

評測數據成為機房報告的基礎；你的貢獻會被明確致謝。

## ❓ 常見問題

**Q: 我不會編程，能否貢獻？**
A: 當然可以！貢獻方式包括：標注評測集、回報 Bug、改進文檔、幫忙翻譯。

**Q: 我想在自己的國家推廣，可以嗎？**
A: 歡迎！請先聯繫我們討論。我們可以幫你適配規則庫、分享經驗。

**Q: 代碼修改後，什麼時候會被併入主分支？**
A: 通常 1–2 週內進行 Code Review。複雜變更可能需要更長討論。

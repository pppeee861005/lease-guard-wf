### 詳細驗收步驟（以項目1為例）

**步驟1：準備**（1分鐘）
```
1. 打開官方 PDF，找到「項目1：審閱期」的原文
2. 打開規則庫 JSON，找到 "id": "1"
3. 在驗收清單上標記「進行中」
```

**步驟2：驗收 Original 欄位**（2分鐘）
```
1. 複製官方 PDF 的項目1原文
2. 複製 JSON 中的 original 欄位
3. 逐字比對，檢查：
   ✅ 是否完全相同？
   ✅ 有無多字或少字？
   ✅ 標點符號是否相同？
   ✅ 段落結構是否相同？

4. 結果：
   ✅ 相同 → 打勾，繼續
   ❌ 不同 → 記錄到 D3_disputed.md，繼續（不修改，人類簽名人決定）
```

**步驟3：驗收 Plain 欄位**（2分鐘）
```
1. 讀 JSON 中的 plain 欄位
2. 自測檢查：
   ✅ 我看得懂嗎？
   ✅ 有沒有法律術語（「定著物」「動產」「不可移動」）？
   ✅ 是否 1-3 句簡潔？
   ✅ 有沒有具體例子？
   ✅ 有沒有「AI起草，待驗收」標記？

3. 結果：
   ✅ 符合 → 打勾，繼續
   ⚠️ 有問題 → 記錄到 D3_disputed.md，繼續
```

**步驟4：驗收 Violations 欄位**（2分鐘）
```
1. 檢查 violations 陣列：
   ✅ 樣態數：3-5 個？
   ✅ 每個樣態有 pattern + example + severity_flag？
   ✅ example 有具體數字或案例？
   ✅ 邏輯是否合理？

2. 結果：
   ✅ 符合 → 打勾，繼續
   ⚠️ 有問題 → 記錄到 D3_disputed.md，繼續
```

**步驟5：驗收 Severity 欄位**（1分鐘）
```
1. 檢查判定邏輯：
   ✅ 符合決策流程嗎？（見下方決策樹）
   ✅ 與 violations 的 severity_flag 一致嗎？
   ✅ 與該項性質相符嗎？

2. 結果：
   ✅ 符合 → 打勾，繼續
   ⚠️ 有疑義 → 記錄到 D3_disputed.md，繼續
```

**步驟6：完成標記**（1分鐘）
```
1. 在驗收清單上標記「✅ 完成」
2. 如果沒有問題 → JSON 中移除「AI起草，待驗收」標記
3. Git commit：git commit -m "Verify batch1-item-01-審閱期"
4. 進入下一項
```

**總耗時**：6-10 分鐘/項（含各種情況）

### Severity 決策樹（快速判斷）

```
該項涉及「金錢損失」嗎？
├─ 是 → 計算年損失金額
│  ├─ ≥$12,000/年 → 「高」✅
│  ├─ $3,600-12,000/年 → 「中」✅
│  └─ <$3,600/年 → 「低」✅
│
└─ 否 → 該項涉及「居住權」嗎？
   ├─ 是（例如終止權限制、安全威脅）→ 「高」✅
   └─ 否 → 該項涉及「能否事後救濟」嗎？
      ├─ 難救濟（例如不可逆的決定） → 「高」✅
      ├─ 可救濟（例如能走法律程序） → 「中」✅
      └─ 程序瑕疵（例如無實質損害） → 「低」✅
```

**驗收快速檢查**：
```

Violations 應該包含：
「按最高級距計費（例如 400 度按 $6.12 元計），
 導致年度超收 $11,388 元」

Severity 應該是：「高」（年損失超過 $12,000 邊界）
```

#### ⭐ 項目18（承租人終止權）- 114年新規重點

**預期問題**：
- Original 是否清楚標記出 114 年 4 月 18 日的修正？
- Original 是否列舉了六種新的終止情形？
- Plain 是否簡潔說明「門檻降低了」？
- Violations 是否涵蓋舊法違規案例（禁止提前終止、高額違約金）？

**驗收快速檢查**：
```
Plain 應該能說出：
「114 年新規，房客如果遇到漏水、霉菌等危害健康的問題，
 或房東超過 30 天不修繕，可以提前終止租約搬走。
 這是重大改善，過去門檻太高。」

Violations 應該包含：
「舊法違規：合約禁止承租人提前離開」
「舊法違規：要求高額違約金」
「新規補充：繼承人也可以終止租約」

Severity 應該是：「高」（居住權威脅）
```

## ⚠️ 常見問題與快速應對

### Q1：驗收時發現 Original 有錯誤，怎麼辦？

**A**：
1. **不要自己修改**（Original 是零容忍的）
2. 記錄到 `D3_disputed.md`
3. 繼續驗收下一項（驗收流程不停頓）
4. 晚間週中覆盤時決定是否修改

### Q2：Plain 欄位不夠「易讀」，可以改嗎？

**A**：
1. 記錄問題到 `D3_disputed.md`
2. 繼續驗收（不停頓）
3. 如果問題很明顯（例如全是法律術語），可以記為「打回」
4. 晚間週中覆盤時決定修改方案

### Q3：驗收時間不夠，怎麼辦？

**A**：
1. 優先完成「重點項」（項目6、18）
2. 其他項目順延至 D4 上午
3. 記錄進度到 `D3_執行計畫_20260613.md`
4. 保持「驗收不可趕工」原則，寧可延遲也要保質量

### Q4：Agent 生成的 Batch2 品質低，怎麼辦？

**A**：
1. 不要等，立即啟動「Prompt 校準」流程
2. 修改 `D3_Batch2_Agent_Prompt.md` 的相應部分
3. 重新執行 Agent
4. 記錄調整原因到 `D3_執行計畫_20260613.md`

### Q5：疑義項太多（>5個），影響進度怎麼辦？

**A**：
1. 優先解決「高風險」疑義項
2. 其他疑義項延至 D5 專項討論
3. 不要試圖在驗收時「解決」疑義
4. 記錄清楚，便於後續追蹤

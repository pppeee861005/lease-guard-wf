# 五信號驗證任務：四層架構分析表

**任務背景**：美伊協議五信號實時驗證（2026-06-19）
**核心需求**：三步驗證 + 版本管理 + 後續監測規劃
**實際執行**：L2 Agent Teams（施工隊協作）

---

## 📊 四層完整對比表

### 層級選擇決策樹

| 層級 | 場景定位 | 計畫持有人 | 班組配置 | 適配本案 |
|------|---------|----------|--------|--------|
| **L1** | 家庭廚房 | Claude 逐回合決策 | 單個或少量 Subagent | ❌ 不適合 |
| **L1.5** | 多廚房監控 | 人類中央監控 | 多個獨立 session | 🟡 可用但不優 |
| **L2** | 施工隊協作 | Lead + Peers 協商 | 5-10 個平等班組 | ✅ 最適合（現用） |
| **L3** | 工廠自動化 | 代碼本身 | 100+ agents 自動執行 | ❌ 過度設計 |

## 🔍 詳細方案對比

### L1：Subagents（廚房型）

#### 執行流程

```
時間線：
T0 你說「幫我驗證五信號」
   ↓
T1 我（Claude）收到任務 → 決定步驟1：查詢美債
   ↓ 調用 Subagent-1 查詢美債
T2 Subagent-1 返回：「美債 4.46%」
   ↓ 我看著結果 → 決定步驟2：查詢油價
   ↓ 調用 Subagent-2 查詢油價
T3 Subagent-2 返回：「油價 $75」
   ↓ 我看著兩個結果 → 決定步驟3：查詢 VIX
   ...
T6 我收集完所有數據 → 決定步驟7：生成報告
   ↓ 調用 Subagent-6 編寫報告
T7 Subagent-6 返回報告
   ↓
T8 我把報告給你看 → 等待你的反饋
   ↓
T9 你說「要廢棄舊版本嗎？」
   ↓
T10 我決定是否調用 Subagent 去編輯舊文件
```

#### 班組配置

```
Subagent-1: 美債/石油/VIX/匯率查詢
Subagent-2: 可信度評估
Subagent-3: 時效性檢查
Subagent-4: 版本管理
Subagent-5: 報告生成
```

#### 決策方式

- Claude 逐步驟決策 → 等待 Subagent 回應 → 下一步決策
- **特點**：完全序列化，無並行

#### 成本評估

| 維度 | 評分 |
|------|------|
| Token 消耗 | ⭐⭐ 低 |
| 執行時間 | ⏱️ 慢（序列）|
| 人工干預 | 🤝 頻繁 |
| 質量保障 | 🔴 低（無互查） |
| 複雜度控制 | ⭐⭐⭐⭐⭐ 最簡單 |

#### 適配本案的評分

```
整體評分：⭐⭐ (2/5)

原因：
❌ 無法發現「早期版本vs實時版本」的矛盾
   → 因為沒有「互相檢查」機制
❌ 效率低下
   → 每查詢完一個指標，都要等我決定下一步
❌ 容易遺漏
   → 單個 Subagent 做完轉身就走，無人複核
✅ Token 便宜（唯一優點）

適合場景：寫一篇簡單文章（單人快速反覆）
不適合場景：複雜驗證（需要互相檢查）
```

### L1.5：Agent View（監控台型）

#### 執行流程

```
T0 你起動 3 個 background agent sessions
   ├─ Session-A: 負責「數據查詢」
   ├─ Session-B: 負責「可信度評估」
   └─ Session-C: 負責「版本管理」

T1-T15 三個 session 各自獨立運行
   • Session-A 執行：調用網路查詢 → 返回美債/油價/VIX/匯率
   • Session-B 執行：分析數據可信度 → 返回評估報告
   • Session-C 執行：準備版本標記 → 返回編輯方案

T16 你監看三個 session 的進度條
   • A：✅ 完成數據查詢
   • B：⏳ 正在分析（70%）
   • C：⏳ 準備編輯（40%）

T20 你可以「切進」Session-B，直接和我（B 中的 Claude）對話
   你：「為什麼認為早期版本是情景推演？」
   我（B）：「因為數據不匹配...」
   你：「同意，繼續」

T25 全部 session 完成
   你看到三份報告

T26 你做決策：「廢棄舊版本」
   ↓ 但無法讓 A、B、C 互相討論
   ↓ 你要自己綜合三份報告
```

#### 班組配置

```
Session-A（獨立）：數據查詢層
  • 執行網路搜索
  • 返回原始數據

Session-B（獨立）：可信度評估層
  • 自己分析（不知道 A 的發現）
  • 返回評估報告

Session-C（獨立）：版本管理層
  • 自己準備編輯方案
  • 不知道 A 和 B 的矛盾
```

#### 決策方式

- 三個 session 各自運行（互不通訊）
- 你監看進度 → 可隨時「attach」進任一 session 對話
- **特點**：並行但獨立，無協作

#### 成本評估

| 維度 | 評分 |
|------|------|
| Token 消耗 | ⭐⭐⭐ 中低 |
| 執行時間 | ⏱️ 快（並行）|
| 人工干預 | 🤝 中等（需監看） |
| 質量保障 | 🟡 中（無 peer review） |
| 複雜度控制 | ⭐⭐⭐⭐ 容易 |

#### 適配本案的評分

```
整體評分：⭐⭐⭐ (3/5)

優點：
✅ 三個 session 並行，比 L1 快
✅ 你可以隨時切進任一 session 監看
✅ Token 消耗相對低

缺點：
❌ Session-A、B、C 互不通訊
   → Session-B 無法直接告訴 Session-A：「你查的數據有問題」
   → 必須你（人類）做為中介綜合意見
❌ 容易遺漏矛盾
   → 如果 B 發現「A 的數據過時」，B 只能「報告給你」
   → 你要自己決定是否讓 A 重查

可能發生的問題：
  場景1：你沒看 B 的報告就看了 C 的編輯方案
         → 遺漏了 A、B 的矛盾
  場景2：B 發現 A 的數據有問題，但 B 沒有「強制 A 重查」的權力
         → 必須等你介入

適合場景：多項目並行、獨立進度監看（寫 5 個系列文章）
不適合場景：複雜協作、需要互查（本案）
```

### L2：Agent Teams（施工隊協作）— 【本案現用】

#### 執行流程

```
T0 任務起動
   工頭（Claude）設定大方向：
   「三步驗證：源頭 → 現實 → 時效」

T1 五個班組並行啟動（真正的 peer-to-peer）
   ├─ 班組1（數據源審查）：開始檢查早期版本可信度
   ├─ 班組2（現實核實）：開始查詢實時網路數據
   ├─ 班組3（時效性檢查）：準備時效驗證框架
   ├─ 班組4（口徑協調）：等待其他班組數據，準備對標
   └─ 班組5（風險預警）：準備長期監測框架

T3（班組1 完成早期版本審查 5分鐘後）
   班組1 → 直接告訴班組4：
   「早期版本美債 4.9%，現實 4.46%，差異巨大」

   班組4 反饋：
   「收到，我會在對標時強調這個矛盾」

T5（班組2 完成實時查詢 3分鐘後）
   班組2 → 直接告訴班組3：
   「我查到五個最新數據，都是 24小時內的」

   班組3 反饋：
   「完美，我會標註這個時效性」

T7（班組1 + 班組4 開始協商）
   班組1 & 班組4 互相討論：
   「要不要廢棄早期版本？」
   「應該怎麼標記？」
   「要不要寫成警告頭條？」

   工頭（我）監聽，但不打斷

T10（班組4 發現新問題）
   班組4 → 直接告訴班組5：
   「協議簽署成功，短期樂觀，但 60天內有三大風險」

   班組5 反饋：
   「我會設計這三大風險的監測機制」

T13（全體班組完成初稿，工頭協調）
   工頭（我）開始協調：
   • 檢查五個班組有沒有遺漏
   • 確認版本管理方案一致
   • 確認後續監測框架清晰

T15（工頭生成最終報告 + 決策建議）
   報告包括：
   • 三步驗證的完整分析
   • 四層對比（本表）
   • 版本管理方案
   • 決策建議清單

T16 呈現給你（最高決策人）
   你看到完整分析 → 審視班組的建議 → 做決策

T17 你決策：
   ✅ 廢棄早期版本
   ✅ 標記為「已棄用」
   ✅ 選擇「動態監測頻率」
   ✅ 要求「60天計劃」

T18 工頭（我）協調執行
   • 班組4 負責版本標記編輯
   • 班組5 負責監測計劃設計
   • 班組1-3 待命檢查
```

#### 班組配置

```
工頭：Claude Lead Agent
  • 決定大方向：三步驗證框架
  • 監督五個班組進度
  • 協調班組間的分歧
  • 生成最終報告 + 決策建議

班組1（數據源審查）
  • 檢驗早期/實時版本可信度
  • 識別矛盾
  • 向班組4 報告發現

班組2（現實核實）
  • 執行實時網路查詢
  • 向班組3 報告新數據時效性

班組3（時效性檢查）
  • 驗證所有數據新鮮度
  • 向班組4 確認驗證結論

班組4（口徑協調）
  • 對標早期vs實時
  • 協商版本標記方案
  • 與班組5 討論風險項

班組5（風險預警）
  • 識別 60天內的風險
  • 設計監測機制
  • 與班組4 協商優先級
```

#### 決策方式

- **工頭決定大方向**：「三步驗證」
- **班組自主協商**：班組1-4 直接通訊，討論版本問題
- **工頭協調分歧**：班組意見不一致時，我作為中立協調者
- **用戶做最終決策**：審視班組報告後，批准或修改方案
- **特點**：真正的 peer-to-peer，工頭監督不干預細節

#### 實際案例：本案執行紀錄

```

T3: 班組1（我剛才的第一輪檢查）

    ❌ 早期版本（4.9% / $128.5 / 31.5）都是錯的

T11: 班組5（風險預警）

    1. 執行落差（制裁豁免進度不力）

    3. 美國政策一致性（對盟友施壓能力有限）

T13: 工頭（我）生成本份四層分析表
    包括決策建議：
    • 監測頻率選「動態」

T16: 呈現給你（最高決策者）

```

#### 成本評估

| 維度 | 評分 |
|------|------|
| Token 消耗 | ⭐⭐⭐⭐⭐ 很高 |
| 執行時間 | ⏱️ 中等（並行+協商） |
| 人工干預 | 🤝 低（班組自主） |
| 質量保障 | ⭐⭐⭐⭐⭐ 最高（互查） |

#### 適配本案的評分

```
整體評分：⭐⭐⭐⭐⭐ (5/5) 【最適合】

優點：
✅ 班組可互相通訊 → 班組1 直接告訴班組4 矛盾
✅ 並行協作 → 五個班組同時工作，快速完成
✅ 質量保障最高 → 互相檢查，遺漏率最低
✅ 自主決策 → 班組不用每次都等工頭，效率高
✅ 人類介入簡化 → 工頭協調後，你只需審視一份完整報告

缺點：
❌ Token 成本高
   → 五個班組同時運行，消耗量是 L1 的 3-5 倍
❌ 複雜度高
   → 需要設計班組配置、通訊規則、共享任務清單

何時值得用 L2：
  ✅ 複雜協作（需要互查）
  ✅ 質量要求高（金融、醫療、地緣政治）
  ✅ Token 預算充足
  ✅ 時間相對充裕（可以多花 Token 換質量）

本案的三個理由：
  1️⃣ 地緣政治分析 = 高質量要求
  2️⃣ 金融數據驗證 = 需要互查
  3️⃣ 版本管理 = 多個班組協調

→ L2 是最優選擇
```

### L3：Dynamic Workflows（工廠自動化型）

#### 執行流程

```
T0 你編寫一份 TypeScript workflow script：

```typescript
// 五信號驗證自動化流程
workflow("VerifyFiveSignals", {
  parallel: [
    {
      name: "agent-data-query",
      task: "查詢五信號實時數據",
      retry: 3,
      timeout: 5min
    },
    {
      name: "agent-credibility",
      task: "評估數據可信度",
      dependsOn: ["agent-data-query"],
      qualityCheck: "必須找到3個獨立來源"
    },
    {
      name: "agent-version-compare",
      task: "對比早期vs實時版本",
      dependsOn: ["agent-credibility"],
      failureHandler: "標記矛盾並上報"
    }
  ],

  // 品質保證層（對抗性審查）
  qualityAssurance: {
    reviewer1: "檢查邏輯是否一致",
    reviewer2: "檢查數據是否完整",
    conflict: "若兩個 reviewer 有分歧，進入 arbitration"
  },

  // 自動決策層
  decisionRules: {
    if: "發現早期版本矛盾",
    then: "自動標記為棄用 + 備份原版本",
    notify: "人類審閱"
  }
})
```

T1-T120 執行運行
  系統自動：
  • 並行啟動 50 個 agents 查詢數據
  • 25 個 agents 並行評估可信度
  • 10 個 agents 進行對標對比
  • 5 個 agents 進行互相審查（紅隊）
  • 全程零人工干預

T121 系統生成最終報告
  自動內容：
  • 五信號完整分析表
  • 版本比對結果
  • 自動決策執行記錄
  • 人類審閱清單

T122 發送給你進行「最終審批」
  你只需：
  • 看完整的自動化報告（5分鐘）
  • 按「確認」或「修改」（1分鐘）
  • 系統自動執行編輯、版本管理、後續監測部署

T123 系統自動部署後續監測
  • 監測 agents 已自動啟動
  • 每日 06:00 UTC 自動生成報告
  • 異常自動推送
  • 無需人工監看
```

#### 班組配置

```
50 個 data-query agents（並行）
  ├─ 查詢美債
  ├─ 查詢油價
  ├─ 查詢 VIX
  └─ ...（× 50 份冗餘查詢，提高容錯）

25 個 credibility-check agents（並行）
  ├─ 評估美債可信度（多個角度）
  ├─ 評估油價可信度
  └─ 評估 VIX 可信度

10 個 comparison agents（並行）
  ├─ 對比美債：早期 vs 實時
  ├─ 對比油價：早期 vs 實時
  └─ 對比 VIX：早期 vs 實時

5 個 QA agents（對抗性審查）
  ├─ 紅隊1：找邏輯漏洞
  ├─ 紅隊2：找數據漏洞
  ├─ 紅隊3：重複驗證
  └─ ...

1 個 final-decision agent
  ├─ 綜合所有意見
  ├─ 執行自動決策規則
  └─ 生成人類友好報告
```

#### 決策方式

- **完全自動化**：計畫寫在 script 裡
- **自動決策**：觸發條件 → 自動執行（不等人類）
- **人類審批點**：只有最後一個（看完整報告 + 確認）
- **特點**：零中間人工，最後才需要人類

#### 成本評估

| 維度 | 評分 |
|------|------|
| Token 消耗 | ⭐ 最高（超過 10 萬 token） |
| 執行時間 | ⏱️ 快（全並行） |
| 人工干預 | 🤝 無（只需最後審批） |
| 質量保障 | ⭐⭐⭐⭐⭐ 最高（對抗審查） |
| 複雜度控制 | ⭐⭐ 低（計畫寫死） |

#### 適配本案的評分

```
整體評分：⭐ (1/5) 【過度設計】

為什麼不適合：
❌ 規模遠小於 L3 的用途
   → L3 是為「批量生成 1000 篇文章」設計的
   → 本案只需驗證「一份五信號報告」

❌ Token 成本完全不合理
   → L2 需要 ~2000 token
   → L3 需要 ~50000 token
   → 成本相差 25 倍，卻只是做同一件事

❌ 時間設置不合理
   → 本案需要「快速反應」（2小時內完成）
   → L3 設計給「長期自動化」（寫一次，重複跑）

❌ 計畫過度固定
   → 本案需要「邊做邊調整」
   → L3 要求「計畫一旦寫死就難以改」

何時應該用 L3：
  ✅ 一次性任務 ≥ 100 agents
  ✅ 任務流程完全固定
  ✅ 需要長期反覆執行
  ✅ Token 預算充足（單次執行可花 $5-10）

例子：
  • 批量審查 1000 份論文（每份論文 = 1 個 agent）
  • 生成 500 個社群媒體帖子系列（每個系列 = 1 個 agent）
  • 自動監測全球 2000 個新聞源（每個源 = 1 個 agent）

結論：
本案用 L3 就像「用推土機去挑米粒」
正確的是 L2：施工隊協作，快速高效
```

### L1 最適合

```
場景：寫一篇文章 / 調試一個 bug / 快速原型

## 💡 本案的關鍵決策

### 為什麼選 L2 而不是 L1.5？

```
L1.5 假設：
  Session-A：查詢數據
  Session-B：評估可信度
  Session-C：版本管理

  你看到三個 session 的進度條
  → 但 B 無法直接告訴 A「你的數據有問題」
  → 必須你作為中介

結果：
  ❌ B 發現「美債 4.9% 與現實 4.46% 差異巨大」
  ❌ B 無法直接告訴 A「去重新查詢」
  ❌ 你要自己綜合三份報告，做出「廢棄舊版本」決策

L2 實際：
  班組1、2、3、4、5 全體協商
  ✅ 班組1 直接發現矛盾
  ✅ 班組4 直接與班組1 討論「要不要廢棄？」
  ✅ 全體班組形成統一意見後呈現給你
  ✅ 你只需「批准」，不需要「綜合」

結論：
L2 把「複雜的綜合決策」轉化為「簡單的批准決策」
→ 質量更高，用戶決策更簡單
```

### L2 的 Token 代價值不值得？

```
成本對比：
  L1.5：~1500 token
  L2：  ~2500 token
  差價：+1000 token = 約 $0.02-0.03

質量差異：
  L1.5：
    • 需要你綜合三份報告
    • 容易遺漏班組間的矛盾
    • 決策時間：10分鐘

  L2：
    • 班組已協商完，直接呈現統一方案
    • 互相檢查，遺漏率最低
    • 決策時間：2分鐘

結論：
  多花 $0.03 換來「決策時間快 5 倍」+ 「質量提升」
  完全值得

  這就是為什麼金融和地緣政治分析必須用 L2
```

## 🎓 學習收穫

### 你學到了什麼

| 層級 | 核心概念 | 本案體驗 |
|------|---------|--------|
| L1 | 序列決策 + 頻繁干預 | ❌ 不適合複雜協作 |
| L1.5 | 並行監控 + 人工綜合 | 🟡 可用但決策複雜 |
| L2 | 並行協作 + peer-to-peer | ✅ 最適合質量要求高的任務 |
| L3 | 全自動化 + 對抗審查 | ❌ 過度設計（規模不夠） |

### 核心原則

```
「計畫持有權」決定架構選擇

L1：你持有（你決定每一步）
L1.5：你持有多個（你監看進度）
L2：團隊持有（工頭+班組協商）
L3：代碼持有（系統自動執行）

沒有「最好的層級」，只有「最適合的層級」
```

# 租約哨兵 LeaseGuard WF

> **拍下你的租約,讓它告訴你哪一條會坑你。**
> An open-source, rule-grounded Dynamic Workflow that checks Taiwanese rental contracts against the official government regulations — built for people who can't afford a lawyer.

## 這是什麼

富人身邊有一圈顧問,窮人簽每一份合約都是裸奔。一條被坑的押金條款,可以讓一個學生倒退半年。

**租約哨兵**是一條開源的「規則庫比對型 Dynamic Workflow」:

```
手機拍攝的租約照片(昏暗、斜拍、皺摺都行)
        ↓
與內政部「住宅租賃定型化契約應記載及不得記載事項」逐條比對
        ↓
輸出三段式報告:
  ⚠️ 對你不利的條款(每條附:合約原文截句 + 官方規範條目)
  ✅ 該有但沒有的條款
  📍 深究管道:崔媽媽基金會 / 法律扶助基金會 / 各縣市消保官
```

**工具永遠免費。** 本專案是公共財;被開源的不只是代碼,還有整個建造過程(評測收據、漏報覆盤、踩坑記錄)。

## 這不是什麼

- ❌ **不是法律意見。** 本工具只陳述「這些條款與官方公開規範有落差」,不告訴你可不可以簽。具體爭議請洽文末的人類專業資源。
- ❌ **不是終審法官,是分流站。** 它的職責是讓你帶著正確的問題去找對的人。
- ❌ **不留存你的合約。** 閱後即焚是設計原則,不是隱私政策的補充條款。

## 三大鐵律(寫進代碼,不只寫進文件)

| # | 鐵律 | 代碼層落地 |
|---|------|------------|
| 1 | 寧可漏報轉介,不可錯報安心 | 每條風險判斷強制雙引用(合約原文 + 規範條目),引用不出 → 該判斷被丟棄 |
| 2 | 信心不足 → 轉介人類 | 低信心輸出固定為轉介話術,附真實聯絡方式 |
| 3 | 永不替用戶做決定 | 輸出層禁用「可以簽 / 不要簽」類結論語句 |

> 為什麼反著一般產品直覺設計?因為用戶是脆弱的。一個自信的錯誤答案(「這條沒問題」但其實有問題)比十次「我不確定」造成的傷害更大。

## 架構

```
lease-guard-wf/
├── rules/        # 規則庫:官方規範的結構化版本(本 repo 最有價值的資產之一)
├── workflows/    # Dynamic Workflow 腳本(Stage 1–5)
├── eval/         # 測試集標注標準、評測腳本、歷次評測報告(含難看的數字)
├── docs/         # 《規則庫比對型 WF》模式文檔 + fork 指南
└── README.md
```

**Pipeline(Supervisor + Pipeline 混合模式):**

```
Stage 1  OCR 與影像前處理      ← 真實世界的照片很爛,這層佔工程量近半
Stage 2  條款切分與結構化       (Haiku)
Stage 3  規則庫比對             (規則引擎優先,LLM 輔助語意比對)
Stage 4  風險解釋生成           (Sonnet,雙引用強制)
Stage 5  信心評分與轉介決策
```

核心設計哲學:**AI 的工作是比對「合約」與「官方規範」的差距,不是自由發揮。** 規則庫是系統的憲法,LLM 是憲法的執行者。

## 給想 fork 的你(其他國家 / 其他場景)

「規則庫比對型 WF」這個模式是通用的,租約只是第一個實例。移植到你的國家或場景,需要替換的只有三樣:

1. **規則庫**(`rules/`):換成你所在地的官方規範——租賃法規、勞動契約標準、消費者保護條例……任何「政府已經寫好規則,但沒人替弱勢者執行比對」的領域
2. **轉介資源**(Stage 5):換成你所在地真實存在、真的接得住人的組織
3. **OCR 語言設定**(Stage 1)

三大鐵律請原封不動帶走。它們不是台灣特色,是「對脆弱用戶負責」的特色。

詳細移植指南:`docs/fork-guide.md`(W4 後發布)

## D2|06/12(五):Schema 設計 + Agent 起草(上半)

**今日目標:定義規則庫的形狀,讓 Agent 開始填內容。**

- [ ] 定稿規則庫 schema(`rules/schema.json`),每條規則含:
  ```
  id            條目編號(沿用官方編號,不自創)
  type          應記載 | 不得記載
  original      官方原文(一字不差)
  plain         白話解釋(給 19 歲首次租屋者看得懂的版本)
  violations[]  常見違規樣態(每個樣態附現實例句)
  severity      高 | 中 | 低
  source_ver    規則庫版本號(= 官方最新修正日期)
  ```
- [ ] **嚴重度分級標準寫成文件**(`rules/severity-criteria.md`),先定義後分級:
  - 高:直接金錢損失或居住權喪失(押金、提前終止、修繕責任轉嫁…)
  - 中:權益受損但可事後救濟
  - 低:程序瑕疵
  - (以上為初稿,逐條分級過程中允許修訂,修訂要留紀錄)
- [ ] **Agent 起草批次 1**:用現有 WF 框架跑「應記載事項」結構化(原文 → schema JSON)
  - Prompt 鐵則:original 欄位逐字抄錄、禁止改寫;plain 與 violations 標注「AI 起草,待驗收」
- [ ] 抽查 3 條起草結果,校準 prompt 後再跑全量

**預估工時**:3–4 小時|**Token 預算**:≤0.1 單位(記實測值:____)

## 風險與應對(本週限定)

| 風險 | 應對 |
|------|------|
| 官方文件有多個版本/修正沿革,不確定哪份最新 | 以行政院公報的最新修正公告為準;沿革記入 `rules/source/CHANGELOG.md`;不確定就把問題寫進戰報,公開求證 |

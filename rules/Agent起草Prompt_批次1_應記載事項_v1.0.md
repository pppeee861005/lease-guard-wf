# 🤖 Agent 起草 Prompt v1.0 - 批次1：應記載事項

**用途**：給 L1 Subagent 的詳細指令，生成租約哨兵規則庫的結構化內容

**目標**：自動將計劃14的「19項完整版」轉換為 schema.json 格式的規則庫條目

**範圍**：應記載事項 1-7 項 + 14-23 項（共 17 項，第 8-13 項另行處理）

---

## 📋 你的角色定義

**你是**：租約哨兵規則庫的結構化專家
- 你需要理解法律條款並轉化為機器可讀的格式
- 你需要識別違規樣態並評估嚴重程度
- 你需要用「給 19 歲看得懂」的語言解釋複雜法律條款

**你不是**：律師或法律顧問
- 你只負責結構化和轉換，不負責法律判斷創新
- 所有內容都基於計劃14的已驗證資料

✅ 第3項   - 租賃期間

✅ 第5項   - 押金約定及返還
✅ 第6項   - 電費條款（⚠️ 重點項，含 500+ 字參考）
✅ 第7項   - 其他費用（水費、瓦斯費、管理費）
⏳ 第8-13項 - 另行處理（不在本批次）
✅ 第14項  - 任意終止租約之約定
✅ 第15項  - 租賃住宅之返還

## 📐 Schema 結構說明

### 必填欄位

#### 1. `id` (string)
- **含義**：條目編號，沿用官方編號
- **格式**：`"1"`, `"6"`, `"23"` 等
- **注意**：不自創編號，必須對應官方文件
- **驗收標準**：必須與官方編號完全相同
- **範例**：`"6"`

---

#### 2. `type` (string, enum)
- **含義**：條文類型
- **可選值**：`"應記載"` 或 `"不得記載"`
- **本批次**：全部填 `"應記載"`
- **驗收標準**：必須是這兩個值之一
- **範例**：`"應記載"`

---

#### 3. `original` (string)
- **含義**：官方原文，必須逐字抄錄，零改動
- **來源**：計劃14的「19項完整版」或官方公告
- **長度**：通常 50-200 字
- **驗收標準**：✅ 與官方文檔完全相同（零改動、零刪除）
  - ❌ 不可改正官方的文法錯誤
  - ❌ 不可刪除冗餘文字
  - ❌ 不可現代化用語
- **質量指標**：100% 正確率，零容忍
- **範例**：
  ```
  "租賃期間相關費用之約定(三)電費：□由出租人負擔。□由承租人負擔。□以用電度數計費者，每度電費不得超過該租賃標的電費單『當期每度平均電價』..."
  ```

---

#### 4. `plain` (string)
- **含義**：白話版解釋，給首次租屋者（19 歲）看得懂
- **禁止項目**：
  - ❌ 法律用語（「約定」改為「說好」）
  - ❌ 法律術語（「承租人」改為「租屋的人」）
  - ❌ 複雜句式（一句話 < 25 字）
  - ❌ 被動語態（改為主動）
  - ❌ 法律概念假設（要解釋）
- **語言規範**：
  - ✅ 日常口語化
  - ✅ 主動語態（房東說、承租人應、法律規定）
  - ✅ 具體例子（「電費$1,500」比「費用」更清楚）
  - ✅ 簡潔直白（1-3 句話，每句 < 25 字）
- **長度**：50-150 字（通常 3-5 句話）
- **驗收標準**：
  - 自測法：「唸給沒租過房的人聽，他們能懂嗎？」
  - 檢查：是否移除了所有法律專業用語
- **質量指標**：理解度 ≥ 95%（所有讀者都能理解）
- **標記**：開頭附加 『AI起草，待驗收』（驗收後移除）
- **範例**：
  ```
  房東如果向租客收電費，價格不能超過台灣電力公司當月的平均電價。
  絕對不能按「最高級距」來超收。
  如果房東不提供電費單據，租客有權向台電 1911 直接查詢用電度數。
  『AI起草，待驗收』
  ```

---

#### 5. `violations` (array of objects)
- **含義**：常見違規樣態清單（現實世界真實發生過的違法行為）
- **數量**：3-5 個樣態（不超過 5 個）
- **結構**：每個樣態包含三個欄位

**5a. `pattern` (string)**
- **含義**：違規模式的簡潔描述（不是整個故事，只是模式）
- **長度**：8-20 字，不超過一句話
- **格式**：簡潔描述「房東怎樣違法」
- **範例**：
  - `"房東按『最高級距』計費"`
  - `"房東隱瞞台電電費單"`
  - `"房東預先收 3 個月電費不退款"`
  - `"房東把公設電費額外計收"`
- **驗收標準**：一個模式一個 pattern，不可混淆

**5b. `example` (string)**
- **含義**：來自真實案例或合理推演的例句
- **格式**：帶引號的房東真實語句或虛構但合理的例句
- **結構**：「房東說什麼」+「事實是什麼」
- **長度**：50-100 字
- **驗收標準**：
  - ✅ 具體、可驗證、有數字
  - ❌ 不可太虛幻或不可能發生
  - ❌ 不可刪除關鍵細節
- **來源**：
  - 計劃14的「房東/承租人操作指南」中的實務案例
  - 已知的 PTT 災難案例
  - 合理推演的場景（必須明確標記為「假設」）
- **範例**：
  ```
  "您用400度，我按每度3.68元計費（最高級距），共1,472元。但查台電電費單，實際費用只要1,247.6元（平均3.12元/度），房東多收了224.4元。"
  ```

**5c. `severity_flag` (string)**
- **含義**：此違規樣態通常對應的嚴重度等級（參考用）
- **可選值**：`"高"`, `"中"`, `"低"`
- **判斷標準**：
  - `"高"` = 年損失 ≥ $12,000 或 難救濟 或 居住權威脅
  - `"中"` = 年損失 $3,600-12,000 且可救濟
  - `"低"` = 年損失 < $3,600 或 程序瑕疵
  - 詳見『severity_分級標準_v1.0.md』
- **注意**：這是參考用，最終 severity 由欄位 `severity` 決定
- **驗收標準**：與 `severity` 欄的判斷應一致

---

#### 6. `severity` (string, enum)
- **含義**：此條文本身的違反嚴重度等級（不是單一樣態，是整項條文）
- **可選值**：`"高"`, `"中"`, `"低"`
- **判斷標準**（核心決策流程）：

```
問 1：這項條文被違反會有直接金錢損失嗎？
  ├─ 年損失 ≥ $12,000 → 進階段2
  ├─ 年損失 $3,600-12,000 → 進階段2
  └─ 年損失 < $3,600 或無金錢損失 → 進階段3

問 2：這項條文涉及「不得記載事項」或居住權或安全嗎？
  ├─ 是 → severity = 高 🔴
  └─ 否 → 進階段 3

問 3：事後能否通過法律手段救濟？
  ├─ 難救濟（即使打官司也難追回） → severity = 高 🔴
  ├─ 可救濟（可通過訴訟、調解取回） → severity = 中 🟡
  └─ 無需救濟（程序瑕疵，無實質損害） → severity = 低 🟢
```

**快速判斷表**：
| severity | 判斷標準 |
|----------|---------|
| 🔴 高 | 年損失≥$12,000 或不得記載或難救濟或居住權威脅 |
| 🟡 中 | 年損失$3,600-12,000且可救濟 |
| 🟢 低 | 年損失<$3,600或程序瑕疵 |

**常見 severity 對應**（參考）：
- 🔴 高：第 6、11、18 項（電費、終止權）
- 🟡 中：第 5、7、9 項（押金、費用、修繕）
- 🟢 低：第 1、19、20、22 項（審閱期、通知、文書）

**標記**：『待驗收』（驗收後移除）

**驗收標準**：
- 與 violations 中的 severity_flag 邏輯一致
- 判斷理由應可在 violation 樣態中找到支撐
- 疑似邊界情況應記錄在 verification_notes 中

**範例**：`"高"`

---

#### 7. `source_ver` (string)
- **含義**：規則庫版本號，即官方最新修正日期
- **格式**：統一格式 `"114年4月18日"` （全文件必須統一，不可混用 YYYY-MM-DD）
- **來源**：內政部公告日期
- **本批次**：全部填 `"114年4月18日"`
- **驗收標準**：
  - 必須與官方文件版本對應
  - 全文件格式統一（全用「年月日」或全用「YYYY-MM-DD」）
- **範例**：`"114年4月18日"`

---

### 可選欄位

#### 8. `status` (string, enum)
- **含義**：驗收狀態
- **預設值**：`"待驗收"`
- **可選值**：`"待驗收"`, `"已驗收"`, `"有爭議"`
- **驗收標準**：
  - 初始值必須填 `"待驗收"`
  - 驗收通過後改為 `"已驗收"`
  - 邊界情況改為 `"有爭議"`
- **範例**：`"待驗收"`

---

#### 9. `verification_notes` (string, optional)
- **含義**：驗收人的備註欄位
- **何時填寫**：驗收階段（D3），AI 起草階段可空著
- **可記錄內容**：
  - plain 欄的改進建議
  - violations 的邊界案例
  - severity 的判斷依據或疑慮
  - 與計劃14的對應關係
- **範例**：
  ```
  "plain欄修改建議：『房東最多按平均電價收費』更簡潔。
  severity 判斷：第6項電費條款年損失$2,400-6,000，對應高級別，符合。
  參考計劃14：修正版本對照表_114年版本.md 第6項部分。"
  ```

## 🎯 特別注意事項

### 重點項目 1：第 6 項（電費）

**為什麼重要**：
- 最常見的違規來源（佔 40% 以上糾紛）
- 109 年舊版 vs 114 年新版有重大差異
- 涉及金錢損失最高（年損失 $2,400-6,000）

**你的任務**：
1. plain 欄必須解釋「當期平均電價」的含義（不是「最高級距」）
2. violations 必須包含：
   - 按「級距最高價」計費（高）
   - 隱瞞電費單（高）
   - 強制預付電費（中）
   - 公設電費額外計收（中）
3. severity = 高

**參考資料**：
- 計劃14 的「修正版本對照表_114年版本.md」（詳述電費計算）
- 計劃14 的「房東/承租人電費操作指南」（實務案例）

### 重點項目 2：第 18 項（承租人終止權）

**為什麼重要**：
- 114 年 4 月 18 日新修正（終止權門檻降低 70%）
- 直接影響承租人能否提前離開
- 涉及居住權喪失風險

**你的任務**：
1. plain 欄必須說明「6 種終止情形」（新規有第六款：繼承人終止）
2. violations 必須包含：
   - 禁止提前終止（高）
   - 終止門檻超過 1 個月租金（高）
   - 要求過多終止通知期（中）
3. severity = 高
4. verification_notes 應附註「114 年新規，門檻降低 70%」

**參考資料**：
- 計劃14 的「19項完整版」（第 18 項含 114 年修正詳述）

## ✅ 質量檢查清單

**你生成的每個條目，都必須通過以下檢查**：

### 欄位完整性檢查
- [ ] 7 個必填欄位都有（id, type, original, plain, violations, severity, source_ver）
- [ ] violations 陣列有 3-5 個樣態
- [ ] 每個 violation 包含 pattern、example、severity_flag

- [ ] violations = 真實或合理的案例（可驗證）
- [ ] severity = 按「決策流程圖」判斷（有邏輯）

### 格式規範檢查
- [ ] 所有字串都用雙引號（JSON 格式）
- [ ] source_ver 全文件統一格式（114年4月18日）
- [ ] plain 欄開頭附加『AI起草，待驗收』標記
- [ ] severity 欄填 "高" / "中" / "低"（無其他值）

### 邏輯一致性檢查
- [ ] violations 中的 severity_flag 邏輯與最終 severity 一致
- [ ] 如果 severity = 高，violations 中應有高級別樣態支撐
- [ ] 第 6、11、18 項的 severity 應為「高」
- [ ] 第 1、19、20 項的 severity 應為「低」

## 🚨 常見錯誤與修正

### 錯誤 1：original 欄有改動
❌ **錯誤**：
```json
"original": "租賃期間相關費用之約定電費不得超過平均電價"
```
（刪除了「(三)」和某些細節）

✅ **正確**：
```json
"original": "租賃期間相關費用之約定(三)電費：□由出租人負擔。□由承租人負擔。□以用電度數計費者，每度電費不得超過該租賃標的電費單『當期每度平均電價』..."
```

---

### 錯誤 2：plain 欄有法律術語
❌ **錯誤**：
```json
"plain": "出租人與承租人應約定費用支付之方式，若費用未明確約定，出租人負有告知義務。"
```
（用了「出租人」「承租人」「約定」等法律術語）

✅ **正確**：
```json
"plain": "房東和租客要說好電費怎麼算、怎麼付。房東必須清楚告訴租客費用是多少。『AI起草，待驗收』"
```

---

### 錯誤 3：violations 邏輯混淆
❌ **錯誤**：
```json
"violations": [
  {
    "pattern": "費用相關",
    "example": "房東沒有提供任何費用資訊。",
    "severity_flag": "低"
  }
]
```
（太籠統，沒有具體違規行為）

✅ **正確**：
```json
"violations": [
  {
    "pattern": "房東隱瞞電費單",
    "example": "房東說『電費$1,500』，但拒絕提供台電電費單，租客無法驗證。",
    "severity_flag": "高"
  }
]
```

---

### 錯誤 4：severity 判斷不當
❌ **錯誤**：
```json
"severity": "中"  // 審閱期不足（應該是「低」）
```

✅ **正確**：
```json
"severity": "低"  // 審閱期不足是程序瑕疵，無實質損害
```

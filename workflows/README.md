## 五階段 Pipeline

### Stage 1: OCR 與影像前處理
- **輸入**：租約照片（手機拍、昏暗、斜拍都可）
- **工作**：
  - 影像去噪與角度校正
  - 區域檢測（租約區域 vs. 背景）
  - OCR 識別
- **輸出**：結構化文本
- **技術**：本地 OCR + Claude 精修

### Stage 2: 條款切分與結構化
- **輸入**：Stage 1 的 OCR 文本
- **工作**：
  - 句子級別的切分
  - 條款編號識別
  - 條款類別初判（應記載 / 不得記載 / 中立）
- **輸出**：JSON 格式的條款列表
- **技術**：Claude Haiku

### Stage 3: 規則庫比對
- **輸入**：Stage 2 的條款列表
- **工作**：
  - 逐條與規則庫匹配（應有 + 不該有）
  - 漏項檢測（該有但沒有的條款）
  - 初步風險標記
- **輸出**：匹配結果 + 風險列表
- **技術**：正則表達式 + 向量檢索 (可選)

### Stage 4: 風險解釋生成
- **輸入**：Stage 3 的風險列表
- **工作**：
  - 為每條風險生成解釋（為什麼有風險）
  - **強制雙引用**：
    - 引用 1：合約中的原文截句
    - 引用 2：規範中的相應條目編號
  - 引用驗證：引用不出 → 該判斷被丟棄
- **輸出**：富文本的風險說明
- **技術**：Claude Sonnet

### Stage 5: 信心評分與轉介決策
- **輸入**：Stage 4 的風險說明
- **工作**：
  - 為每條風險打信心分
  - 低信心 (<0.7) → 轉介話術
  - 輸出格式決策（三段式報告）
- **輸出**：最終報告
- **技術**：規則引擎

## 設計原則

1. **AI 的工作是比對，不是決策**
   - Pipeline 的每一步都明確地比對「合約」與「規範」的差距
   - 不進行自由發揮

2. **三大鐵律貫穿全程**
   - ❌ 寧可漏報轉介，不可錯報安心（雙引用強制）
   - ❌ 信心不足 → 轉介人類
   - ❌ 永不替用戶做決定

3. **可重現性**
   - 同樣的合約 + 同樣的規則庫版本 = 同樣的輸出
   - 便於評測與審計

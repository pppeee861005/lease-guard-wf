### L1：Subagents（廚房型）

#### 執行流程

```
時間線：
T0 你說「驗收第6項電費條款」
   ↓
T1 我（Claude）收到任務 → 決定步驟1：讀取 JSON
   ↓ 調用 Subagent-1 讀取 rules_batch1_generated.json
T2 Subagent-1 返回：「第6項內容...」
   ↓ 我看著結果 → 決定步驟2：比對官方 PDF
   ↓ 調用 Subagent-2 讀取官方原文
T3 Subagent-2 返回：「官方原文...」
   ↓ 我比對 → 發現差異 → 決定步驟3：修正
   ...
T6 我完成驗收 → 更新 JSON
   ↓
T7 等待你確認 → 進入下一項
```

```

適合場景：
  ✅ 規則庫驗收（D3-D5）
```

```
T0 你啟動 3 個 background agent sessions
   ├─ Session-A: 負責「批次1驗收」（17項應記載）

T1-T60 三個 session 各自獨立運行
   • Session-A 執行：逐條比對 → 標記待修正項
   • Session-B 執行：Agent 起草 → 生成 JSON
   • Session-C 執行：Agent 起草 → 生成 JSON

T61 你監看三個 session 的進度
   • A：✅ 完成驗收（發現3項需修正）
```

### L2：Agent Teams（施工隊協作）— 【評測期推薦】

#### 執行流程（中心輻射式 hub-and-spoke）⚠️ 實戰校準版

> **重要校準（2026-06-19 實測）**：L2 的班組**不會互相直接通訊**。所有協調由**主控（Claude Lead）顯式編排** —— 班組各自完成工作後**回報主控**，主控收齊後再派下一棒。這是「中心輻射」，不是「點對點協商」。

```
                    主控（Claude Lead，唯一調度中樞）
                              │ 顯式派遣 / 收齊結果 / 決定下一棒
   ┌──────────┬──────────┬────┴─────┬──────────┐
   ▼          ▼          ▼          ▼          ▼
 班組1      班組2      班組3      班組4      班組5
（各自獨立工作，彼此看不見對方，只跟主控回報）

時間線：
T0 主控設定大方向：「三層流水 + 交叉驗證」

T1 階段一（真並行）：主控同時派遣 1/2/3
   ├─ 班組1：基本資訊提取 → 回報主控
   ├─ 班組2：應記載稽核 → 回報主控
   └─ 班組3：不得記載掃描 → 回報主控
   （三隊互不通訊，各自把結果交回主控）

T2 主控收齊 1/2/3 結果 → 發現班組2、3 結論有衝突
   主控決定：派班組4 做交叉驗證仲裁
   （注意：不是班組4「主動找」班組2，而是主控把
    班組2 的結論餵給班組4，請它獨立復驗）

T3 班組4：回原文 grep 復驗 → 推翻/維持前隊判定 → 回報主控

T4 主控收齊全部 → 派班組5 彙總
   （主控把 1-4 的結果一起交給班組5）

T5 班組5：產出最終報告 → 回報主控

T6 主控驗證檔案 → 呈現給你（最高決策人）
   你審視 → 批准或要求修正
```

**🔑 與舊版（點對點）的根本差異**：
- ❌ 舊版想像：「班組4 → 直接告訴班組3：判斷有誤」（班組自主對話）
- ✅ 實際運作：「主控把班組3 結論餵給班組4 復驗 → 班組4 回報主控 → 主控決定是否重派」
- **設計含義**：你不需要設計「班組間通訊協議」；你要設計的是**主控的編排邏輯**（誰先跑、誰依賴誰、收齊後派誰）。

#### 班組配置

```
主控：Claude Lead Agent（唯一通訊中樞）
  • 決定大方向：三層流水 + 交叉驗證
  • 顯式派遣每個班組（誰先跑、誰依賴誰）
  • 收齊各班組回報 → 決定下一棒
  • 仲裁班組間的結論衝突
  • 生成最終報告 + 決策建議

班組1（基本資訊層）
  • 提取甲乙方/租期/租金等結構化資訊
  • 輸出 JSON → 【回報主控】

班組2（應記載稽核層）
  • 23 項應記載逐條比對
  • 輸出符合度表 → 【回報主控】

班組3（不得記載稽核層）
  • 10 項禁止條款掃描
  • 輸出違反度表 → 【回報主控】

班組4（交叉驗證層）
  • 主控把 1-3 結論餵入 → 獨立回原文 grep 復驗
  • 有權推翻前隊判定 + 5 大風險區紅線驗證
  • 輸出仲裁裁定 → 【回報主控】

班組5（彙總報告層）
  • 主控把 1-4 結果一起餵入 → 統整
  • 生成最終 Markdown 報告 → 【回報主控】

⚠️ 注意：所有「向班組X」其實都是「回報主控，由主控轉派」。
   班組之間沒有直接通道。
```

#### 適配本案的評分

```
整體評分：⭐⭐⭐⭐⭐ (5/5) 【評測期最適】

優點：
✅ 質量保障最高
   → 班組4 交叉驗證 + 有權推翻前隊判定，遺漏率最低
   → 實證：2026-06-19 班組4 推翻了文件自稱的「100%」，實測 84.8%
✅ 並行加速
   → 階段一（班組1/2/3）真並行，wall-clock 大幅壓縮
✅ 獨立復驗
   → 每隊各自回原文舉證，主控收齊後仲裁，不靠單一視角

缺點：
❌ Token 成本高（實測警示）
   → 2026-06-19 跑「1 份」文字租約 5 班組 = 235,019 tokens
   → 是 L1.5 的約 2-3 倍；25 份規模是【百萬級】token，非萬級
❌ 複雜度集中在主控
   → 班組間不通訊，但主控要設計好「派遣順序 + 依賴關係 + 收齊邏輯」
   → 編排錯了（例如該等齊卻沒等）會導致班組拿到不完整輸入

適合場景：
  ✅ W3 批量評測（25份租約）
  ✅ 需要交叉驗證的高質量任務
  ✅ 漏報率要求 <5% 的嚴格標準
  ❌ 規則庫逐條驗收（過度設計）
```

### L3：Dynamic Workflows（工廠自動化型）

#### 執行流程（純 JavaScript 真實 API）⚠️ 實戰校準版

> **重要校準（2026-06-19 實測）**：舊版把 L3 寫成「TypeScript + 宣告式 config 物件」，**全錯**。真實 Workflow/Ultracode 是**純 JavaScript**（型別註記會解析失敗），用**命令式原語** `agent()` / `parallel()` / `pipeline()`，且必須以 `export const meta` 開頭。沒有 `qualityAssurance`/`decisionRules`/`dependsOn` 這些宣告式鍵 —— 交叉驗證與決策規則是用一般 `for`/`if` 控制流 + `agent(prompt,{schema})` 表達。

```javascript
// 租約哨兵批次評測（真實 Workflow 寫法，純 JS）
export const meta = {
  name: 'LeaseGuardBatch',
  description: '25 份租約批次評測：OCR → 檢測 → 交叉驗證',
  phases: [{ title: 'Scan' }, { title: 'QA' }],
}

phase('Scan')
// pipeline：每份租約獨立流過所有 stage，無 barrier（預設首選）
const results = await pipeline(leases,
  (lease) => agent(`OCR 並結構化：${lease.text}`, { schema: OCR_SCHEMA, model: 'haiku' }),
  (doc)   => agent(`規則庫比對偵測違規：${doc.content}`, { schema: DETECT_SCHEMA, model: 'sonnet' }),
  (det)   => agent(`雙引用檢查 + 風險分級：${det}`, { schema: QA_SCHEMA, model: 'opus' })
)

phase('QA')
// 交叉驗證與決策規則：用一般控制流，不是宣告式鍵
const misses = results.filter(Boolean).filter(r => r.severity === '高')
for (const m of misses) {
  const votes = await parallel([          // 對抗式交叉驗證（多個懷疑者）
    () => agent(`試圖推翻此高風險判斷：${m}`, { schema: VERDICT_SCHEMA }),
    () => agent(`試圖推翻此高風險判斷：${m}`, { schema: VERDICT_SCHEMA }),
  ])
  if (votes.filter(v => v.refuted).length >= 1) log(`衝突 → 進入人工審核：${m.id}`)
}
const missRate = misses.length / results.length
if (missRate > 0.05) log('漏報率 > 5% → 建議進入修復週')   // decisionRules = if/條件，不是宣告式鍵

return { results, missRate }
```

**真實併發**：上述 `parallel`/`pipeline` 即使丟 25+ 份，**同時最多約 16 個 agent 跑**，其餘自動排隊（非「100+ 同時」）。
**真實成本**：以實測單份 235k tokens 估，25 份批次屬**百萬級**（≈6–11M tokens），非舊版的「~30000」。

#### 適配本案的評分

```
整體評分：⭐⭐⭐⭐ (4/5) 【W3 即可用，宜用小型 workflow】

✅ 適合的理由（校準後上修）：
✅ 可 scale down
   → Ultracode 沒有「最小規模門檻」，跑 3-7 個 agent 的小 workflow 完全合理
   → 實證：2026-06-19 我們就用一個「7 agent」workflow 審查了這份表的 L3 段
✅ 25 份正是 pipeline 典型用例
   → OCR → 檢測 → QA 三段固定流水，每份獨立流過 = pipeline(items, ...stages)
   → 單次呼叫上限 4096 items，25 份遠在範圍內
✅ 兼得併發加速 + 結構化輸出
   → schema 強制結構化、確定性控制流，比手動 L2 更可複現

⚠️ 唯一仍站得住的限制（降格為「程度」，非「能不能用」）：
⚠️ 規則庫仍在迭代
   → 開發期需頻繁改腳本 + 人工驗收
   → 所以宜用「小型 workflow」，而非「零人工干預的大工廠寫法」
   → 該砍的是「過度自動化的寫法」，不是「Ultracode 工具本身」

📌 舊版三個錯誤前提（已推翻）：
   ❌「L3 為 1000+ 租約設計、25 份規模不夠」→ 無最小門檻，可縮放
   ❌「Token 比 L2 差 6 倍（~30000）」→ 實為百萬級，且與本文成本圖矛盾
   ❌「100+ agents 同時自動執行」→ 同時併發約 16，其餘排隊

結論（校準後）：
本案現階段**就該用** L3 的「小型 workflow」—— 不是推土機，是「剛好大小的工具」。
全自動大工廠（零人工干預）才留給 V2.0 流程固定後。
```

## 📈 成本對比圖（2026-06-19 實測校準）

> **🔬 實測錨點**：1 份「文字」租約 × 5 班組（L2，無 OCR）＝ **235,019 tokens**。
> 以下 25 份數字皆以此錨點外推；含 OCR 時單份再加 ~120k–150k（依定義書，OCR 最耗）。
> ⚠️ **舊版的「萬級 token」估算錯了約 100–400 倍，且與本專案《定義書》的「單份 ~30 萬 token」自相矛盾。已全部修正為百萬級。**

### Token 消耗對比（25 份租約評測，含 OCR）

```

序列只是「慢」，不是「更耗 token」。舊版把 L1 畫成最高 token 是混淆了時間與成本。

L1（逐條，無5隊冗餘，無互查）：  ███████          ~2.5–4M token（最省，但最慢）

L2（5隊含交叉驗證+彙總）：        █████████████    ~6–7.5M token  ← 評測期推薦
L3（全自動+重試+對抗審查）：      ██████████████████ ~9–11M token（最貴）

單份錨點：L2 文字租約實測 235k／份 → ×25 ≈ 5.9M（無OCR）；含OCR ≈ 7.5M
```

## 🎯 本案分階段架構選擇

```
推薦層級：L1.5（監控台型）

原因：
✅ 三個批次可並行生成
✅ 需要頻繁人工介入驗收
✅ 流程還在迭代，需要靈活調整

執行方式：
├─ Session-A: 批次1驗收（17項）
├─ Session-B: 批次2生成（6項）
└─ Session-C: 批次3生成（10項）
```

### 階段二：批量評測（W3）

```
推薦層級：L2（施工隊協作）

原因：
✅ 25份租約需要並行處理
✅ 漏報率要求 <5%，需要交叉驗證
```

```
❌ 流程還在變化
   → L2 需要「明確的協作規則」
   → W2 還在探索最佳流程

結論：
W2 用 L1.5 = 「快速迭代」
```

# 租約哨兵：四層架構分析表

**任務背景**：租約合規檢測系統開發（2026-06-19）
**核心需求**：規則庫建構 + 批量租約檢測 + 風險報告生成
**推薦架構**：L1.5 → L2 混合模式（開發期 L1.5，評測期 L2）

---

## 📊 四層完整對比表

### 層級選擇決策樹

| 層級 | 場景定位 | 計畫持有人 | 班組配置 | 適配本案 |
|------|---------|----------|--------|--------|
| **L1** | 家庭廚房 | Claude 逐回合決策 | 單個或少量 Subagent | 🟡 規則庫驗收可用 |
| **L1.5** | 多廚房監控 | 人類中央監控 | 多個獨立 session | ✅ 開發期最適（現用） |

## 🔍 詳細方案對比

### L1：Subagents（廚房型）

#### 執行流程

```
時間線：
T0 你說「驗收第6項電費條款」
   ↓
T1 我（Claude）收到任務 → 決定步驟1：讀取 JSON
   ↓ 調用 Subagent-1 讀取 rules_batch1_generated.json
T2 Subagent-1 返回：「第6項內容...」
   ↓ 我看著結果 → 決定步驟2：比對官方 PDF
   ↓ 調用 Subagent-2 讀取官方原文
T3 Subagent-2 返回：「官方原文...」
   ↓ 我比對 → 發現差異 → 決定步驟3：修正
   ...
T6 我完成驗收 → 更新 JSON
   ↓
T7 等待你確認 → 進入下一項
```

#### 適配本案的評分

```
整體評分：⭐⭐⭐ (3/5)

優點：
✅ 規則庫逐條驗收非常合適
   → 每次只處理一條，需要人類確認
✅ Token 消耗低
   → 逐條驗收不需要大量並行
✅ 質量控制直接
   → 每條都經過你的眼睛

缺點：
❌ 批量租約檢測時效率低
   → 25份租約逐份處理太慢
❌ 無法發現跨條款的關聯問題
   → 例如「第6項電費」和「第7項其他費用」的邏輯衝突

適合場景：
  ✅ 規則庫驗收（D3-D5）
  ✅ 單條規則修正
  ❌ 批量租約評測（W3）
```

### L1.5：Agent View（監控台型）— 【開發期現用】

#### 執行流程

```
T0 你啟動 3 個 background agent sessions
   ├─ Session-A: 負責「批次1驗收」（17項應記載）
   ├─ Session-B: 負責「批次2生成」（6項）
   └─ Session-C: 負責「批次3生成」（10項不得記載）

T1-T60 三個 session 各自獨立運行
   • Session-A 執行：逐條比對 → 標記待修正項
   • Session-B 執行：Agent 起草 → 生成 JSON
   • Session-C 執行：Agent 起草 → 生成 JSON

T61 你監看三個 session 的進度
   • A：✅ 完成驗收（發現3項需修正）
   • B：✅ 完成生成（6項）
   • C：⏳ 進行中（70%）

T70 你切進 Session-A，處理修正建議
   你：「第6項的 plain 描述要更白話」
   我（A）：「已修正：『房東最多按平均電價收費...』」

T80 全部 session 完成
   你看到三份報告 + 合併後的規則庫
```

#### 班組配置

```
Session-A（獨立）：批次1驗收
  • 17項應記載逐條比對
  • 輸出：修正清單 + 驗收報告

Session-B（獨立）：批次2生成
  • 第8-13項 Agent 起草
  • 輸出：rules_batch2_generated.json

Session-C（獨立）：批次3生成
  • 不得記載事項10項
  • 輸出：rules_batch3_generated.json
```

#### 適配本案的評分

```
整體評分：⭐⭐⭐⭐ (4/5) 【開發期最適】

優點：
✅ 三個批次可並行生成
   → 批次1驗收 + 批次2生成 + 批次3生成 同時進行
✅ 你可以隨時切進任一 session 監看
   → 發現問題立即修正
✅ Token 消耗合理
   → 比 L1 快3倍，成本只增加50%
✅ 獨立性強
   → 一個 session 出問題不影響其他

缺點：
❌ Session 間無法互相通訊
   → 批次2生成時無法參考批次1驗收的修正
   → 必須你手動同步
❌ 批量租約檢測時缺乏質量互查
   → 25份租約各自獨立檢查，無人複核

適合場景：
  ✅ 規則庫分批生成（D3-D5）
  ✅ 多任務並行開發
  🟡 小規模評測（5-10份）
  ❌ 大規模評測（25份+需要互查）
```

### L2：Agent Teams（施工隊協作）— 【評測期推薦】

#### 執行流程

```
T0 任務啟動：25份租約完整評測
   工頭（Claude）設定大方向：
   「三層並行 + 質量互查」

T1 五個班組並行啟動
   ├─ 班組1（OCR層）：5份租約 OCR + 結構化
   ├─ 班組2（OCR層）：5份租約 OCR + 結構化
   ├─ 班組3（檢測層）：等待OCR完成，開始規則比對
   ├─ 班組4（質量層）：等待檢測完成，進行交叉驗證
   └─ 班組5（報告層）：準備報告模板 + 統計框架

T10（班組1、2 完成 OCR）
   班組1 → 直接告訴班組3：
   「5份租約已結構化，發現2份照片模糊需人工確認」

   班組3 反饋：
   「收到，模糊部分先標記為『低信心』」

T20（班組3 完成第一輪檢測）
   班組3 → 直接告訴班組4：
   「檢測完成，發現15個高風險條款、28個中風險條款」

   班組4 開始交叉驗證：
   「我會抽查5個高風險條款，確認引用準確」

T25（班組4 發現問題）
   班組4 → 直接告訴班組3：
   「租約#7的第6項判斷有誤，電費計算方式理解錯誤」

   班組3 反饋：
   「已修正，重新跑租約#7」

T30（全體班組完成初稿，工頭協調）
   工頭（我）開始協調：
   • 檢查五個班組有沒有遺漏
   • 確認漏報率/誤報率計算
   • 生成機房報告草稿

T35（工頭生成最終報告）
   報告包括：
   • 25份租約完整評測結果
   • 漏報率/誤報率統計
   • 每個漏報的根因分析
   • 並行加速效果（vs 順序）

T36 呈現給你（最高決策人）
   你審視 → 批准或要求修正
```

#### 班組配置

```
工頭：Claude Lead Agent
  • 決定大方向：三層並行 + 質量互查
  • 監督五個班組進度
  • 協調班組間的分歧
  • 生成最終報告 + 決策建議

班組1-2（OCR層）
  • 並行處理租約照片
  • 結構化輸出
  • 向班組3 報告完成狀態

班組3（檢測層）
  • 執行規則庫比對
  • 生成風險判斷
  • 向班組4 提交檢測結果

班組4（質量層）
  • 交叉驗證高風險判斷
  • 檢查引用準確性
  • 向班組3 反饋問題

班組5（報告層）
  • 統計漏報率/誤報率
  • 生成機房報告
  • 與班組4 確認數據
```

#### 適配本案的評分

```
整體評分：⭐⭐⭐⭐⭐ (5/5) 【評測期最適】

優點：
✅ 班組可互相通訊
   → 班組4 發現問題，直接告訴班組3 修正
✅ 質量保障最高
   → 交叉驗證，遺漏率最低
✅ 並行加速
   → 25份租約預期 4-5x 加速（vs 順序）
✅ 自主協商
   → 班組不用每次都等工頭，效率高

缺點：
❌ Token 成本高
   → 五個班組同時運行，消耗量是 L1.5 的 2-3 倍
❌ 複雜度高
   → 需要設計班組配置、通訊規則

適合場景：
  ✅ W3 批量評測（25份租約）
  ✅ 需要交叉驗證的高質量任務
  ✅ 漏報率要求 <5% 的嚴格標準
  ❌ 規則庫逐條驗收（過度設計）
```

### L3：Dynamic Workflows（工廠自動化型）

#### 執行流程

```
T0 你編寫一份 TypeScript workflow script：

```typescript
// 租約哨兵自動化流程
workflow("LeaseGuardBatch", {
  parallel: [
    {
      name: "ocr-agents",
      count: 25,  // 25個租約，25個OCR agent
      task: "執行OCR並結構化",
      retry: 3,
      timeout: 5min
    },
    {
      name: "detection-agents",
      count: 25,
      task: "執行規則庫比對",
      dependsOn: ["ocr-agents"],
      qualityCheck: "必須完成雙引用"
    }
  ],

  // 品質保證層（對抗性審查）
  qualityAssurance: {
    reviewer1: "檢查引用準確性",
    reviewer2: "檢查漏報風險",
    conflict: "進入人工審核"
  },

  // 自動決策層
  decisionRules: {
    if: "漏報率 > 5%",
    then: "自動進入修復週",
    notify: "人類審閱"
  }
})
```

T1-T60 系統自動執行
  • 25個 OCR agents 並行
  • 25個檢測 agents 並行
  • 5個 QA agents 交叉驗證
  • 零人工干預

T61 系統生成最終報告
  • 自動計算漏報率/誤報率
  • 自動決策是否進入修復週
  • 人類審閱清單
```

#### 適配本案的評分

```
整體評分：⭐⭐ (2/5) 【過度設計】

為什麼不適合：
❌ 規模不夠大
   → L3 是為「批量處理 1000+ 租約」設計的
   → 本案只需處理 25 份租約

❌ 流程還在迭代
   → 規則庫還在驗收，流程未固定
   → L3 要求「計畫一旦寫死就難以改」

❌ Token 成本不合理
   → L2 需要 ~5000 token
   → L3 需要 ~30000 token
   → 成本相差 6 倍

❌ 開發階段不適合
   → 本案處於 W2 開發期，需要頻繁調整
   → L3 適合成熟期的長期自動化

何時應該用 L3：
  ✅ 上線後每日自動監測（V2.0）
  ✅ 批量處理 100+ 份租約
  ✅ 流程完全固定，不需調整

結論：
本案現階段用 L3 就像「用推土機蓋狗屋」
等 V2.0 上線後，可考慮 L3 做每日自動監測
```

## 💡 關鍵決策：為什麼 W2 用 L1.5 而非 L2？

```
W2 的任務特性：
• 規則庫驗收（需逐條人工確認）
• 批次生成（三個批次獨立）
• 頻繁迭代（發現問題立即修正）

L1.5 優勢：
✅ 三個 session 獨立，一個出問題不影響其他
✅ 你可以隨時切進任一 session
✅ Token 成本低（比 L2 省 30-50%）
✅ 靈活度高（隨時調整方向）

L2 不適合的原因：
❌ 班組協作的開銷大於收益
   → 規則庫驗收本質是「人工逐條確認」
   → 不需要班組互相通訊
❌ 流程還在變化
   → L2 需要「明確的協作規則」
   → W2 還在探索最佳流程

結論：
W2 用 L1.5 = 「快速迭代」
W3 用 L2 = 「質量保障」
```

## 📋 決策速查表

### 選層級時問自己：

```
Q1：任務是「驗收」還是「評測」？
   驗收（逐條確認）→ L1 或 L1.5
   評測（批量檢查）→ L2

Q2：需要交叉驗證嗎？
   否 → L1.5
   是 → L2

Q3：流程固定了嗎？
   還在迭代 → L1.5
   已經固定 → L2 或 L3

Q4：漏報率要求多嚴格？
   <10%（寬鬆）→ L1.5
   <5%（嚴格）→ L2

Q5：是否需要長期自動化？
   否（開發期）→ L1.5 或 L2
   是（上線後）→ L3
```

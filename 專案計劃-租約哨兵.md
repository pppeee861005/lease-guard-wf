# 專案計劃:租約哨兵(LeaseGuard WF)

> **一句話定義**:一條開源的「規則庫比對型 Dynamic Workflow」,把手機拍攝的租約照片,比對內政部官方租賃規範,輸出風險條款警示與人類資源轉介。
>
> **專案性質**:公共財(工具免費開源)+ 方法產品(建造過程即內容,模式可被 fork)
>
> **版本**:v1.0|**制定日期**:2026-06-11|**計劃週期**:30 天(2026-06-11 ~ 2026-07-10)

---

## 一、為什麼做(不可妥協的初衷)

- 窮人做重大決策時沒有顧問。租約是台灣年輕人與弱勢者最高頻、傷害最大的「裸奔決策」之一。
- 一次被坑的押金或惡意條款 = 一個家庭或學生倒退數月到數年。這是「可避免的貧困」。
- AI 第一次讓「每人一個顧問」技術上可行。缺的是把它送到最後一哩、且**不造成二次傷害**的人。
- 對 新人類聯盟 / L3 定位的意義:這是 stock-swarm 證明不了的正當性——框架能解決真實世界的真問題。

## 二、產品定義

### 2.1 用戶旅程(最終形態)
```
加 LINE 好友 → 拍合約每一頁上傳 → 約 1 分鐘後收到三段話:
  ⚠️ 對你不利的條款(每條附:合約原文截句 + 對應官方規範條目)
  ✅ 該有但沒有的條款
  📍 深究管道:崔媽媽基金會 / 法律扶助基金會 / 各縣市消保官(附聯絡方式)
```
零註冊、零學習成本、零術語、零收費。

### 2.2 本期(30 天)只做到
核心 pipeline + 評測報告 + 5 位真人測試。**LINE 介面在第四週評估後才決定是否啟動。**

### 2.3 明確不做(Out of Scope)
- ❌ 不輸出「可以簽 / 不要簽」的結論性建議
- ❌ 不處理債務、保險、勞動契約(未來場景,本期不碰)
- ❌ 不做帳號系統、不留存用戶合約(閱後即焚原則,設計時內建)
- ❌ 不向用戶收費,永遠

## 三、三大設計鐵律(寫進代碼,不只寫進文件)

| # | 鐵律 | 落地機制 |
|---|------|----------|
| 1 | **寧可漏報轉介,不可錯報安心** | 任何風險判斷必須同時引用「合約原文精確字句」+「官方規範具體條目」,引用不出來 → 該判斷禁止輸出 |
| 2 | **信心不足 → 轉介人類** | 標準輸出話術:「這條我看不準,建議帶著合約洽詢崔媽媽基金會(電話)」。轉介是核心功能,不是 fallback |
| 3 | **分流站,不是終審法官** | 輸出永遠是「這幾條與官方規範有落差,簽之前先弄清楚」,絕不替用戶做決定 |

## 四、技術架構

### 4.1 Pipeline(Dynamic Workflow,Supervisor + Pipeline 混合模式)
```
[輸入] 手機照片(預設:昏暗、斜拍、皺摺)
   ↓
Stage 1  OCR 與影像前處理          ← 預估佔總工程量 40-50%,不可低估
   ↓
Stage 2  條款切分與結構化           (Haiku:速度優先)
   ↓
Stage 3  規則庫比對                 (規則引擎優先,LLM 輔助語意比對)
   ↓
Stage 4  風險解釋生成               (Sonnet:每條強制附雙引用,違者丟棄)
   ↓
Stage 5  信心評分與轉介決策          (低信心 → 轉介話術;高信心 → 警示輸出)
   ↓
[輸出] 三段式報告(⚠️ / ✅ / 📍)
```

### 4.2 規則庫
- 來源:內政部「住宅租賃定型化契約應記載及不得記載事項」(公開、權威、條列式)
- 形態:結構化 JSON/YAML,每條規則含:條目編號、原文、白話解釋、常見違規樣態、嚴重度分級
- 原則:**AI 的工作是比對差距,不是自由發揮**。規則庫是系統的憲法。

### 4.3 倉庫結構(GitHub 開源,@houli-fengsheng scope)
```
lease-guard-wf/
├── rules/              # 規則庫(本專案最有價值的資產之一)
├── workflows/          # WF 腳本
├── eval/               # 測試集標注 + 評測腳本 + 歷次評測報告
├── docs/               # 教科書層:「規則庫比對型 WF」模式文檔
└── README.md           # 含三大鐵律、免責聲明、fork 指南(其他國家/場景)
```

## 五、評測方法(上線的唯一門票)

1. **測試集**:25 份以上真實租約(朋友、PTT 租屋版災難案例、崔媽媽公開範例),涵蓋:掃描件 / 手機正拍 / 手機斜拍昏暗照,各佔一定比例
2. **人工標注**:每份合約逐條標注問題條款(ground truth),標注標準寫成文件
3. **指標**:
   - **漏報率**(系統沒抓到的真陷阱)← 首要指標,直接對應用戶傷害
   - **誤報率**(沒問題說成有問題)← 次要,但過高會摧毀信任
   - 引用準確率(輸出的原文截句與規範條目是否真實對應)
   - OCR 端到端成功率(按照片品質分組統計)
4. **誠實原則**:評測報告公開全部數字,包括難看的。漏掉的每一個陷阱單獨寫覆盤。
5. **上線門檻(暫定,第三週依數據校準)**:嚴重度「高」的條款漏報率需低於可向真人交代的水準;達不到 → 第四週改為修復週,不見真人。

## 八、信任與合作

- **不自建信任層**。借用既有信任:大學學生會、宿舍服務中心(灘頭堡)、社區大學、社福團體。
- **W3 結束前寄出第一封合作信給崔媽媽基金會**:定位為「先篩掉八成簡單案件的放大器」,並當面請教法律邊界(什麼構成法律意見、轉介話術如何寫才安全)。法扶基金會為第二接觸對象。
- 法律邊界問題一律以專業人士意見為準,本計劃不自行認定。

## 九、風險登記簿

| 風險 | 等級 | 緩解 |
|------|------|------|
| 錯報安心造成真實傷害 | 🔴 致命 | 三大鐵律 + 雙引用強制 + 漏報率門檻 + 永遠附轉介 |
| OCR 在劣質照片上崩潰 | 🔴 高 | 測試集強制納入劣質照片;前處理獨立成 Stage;必要時輸出「照片看不清,請重拍這幾頁」 |
| 被認定為提供法律意見 | 🟠 中 | 不輸出結論性建議;固定免責聲明;W3 前向崔媽媽/法扶確認話術 |
| 用戶合約隱私外洩 | 🟠 中 | 不留存原件;日誌去識別化;開源 repo 中的測試合約全部去個資 |
| 30 天做不完 | 🟡 低 | 砍範圍不砍鐵律:LINE 可延,評測不可省 |
| 規則庫過時(法規修訂) | 🟡 低 | 規則庫標注版本與抓取日期;每季比對一次官方來源 |

## 十、成功的定義(30 天版)

1. 一條可重現的 pipeline,任何人 clone 後能對一份合約跑出三段式報告
2. 一份公開的、含難看數字的評測報告
3. 5 位真人的真實反饋(哪怕結論是「還不能用」)
4. 「規則庫比對型 WF」作為可 fork 的模式被文檔化

**不在成功定義裡的**:用戶數、訂閱數、任何成長指標。本期只回答一個問題:**這個系統值不值得被一個脆弱的人信任。**

## 📊 案例1：W3 評測（25 份租約）

### 場景描述
- **任務**：檢查 25 份真實租約的合規性
- **Pipeline**：Stage 1-5 完整流程
- **品質要求**：高嚴重度漏報率 < 5%

### 執行模式對比

#### 模式 A：完全順序執行（基準線）
```
租約1（41分） → 租約2（41分） → ... → 租約25（41分）

總耗時 = 25 × 41分 = 1,025 分鐘
       = 17.1 小時
       = 2.1 個工作天（8小時/天）
```

#### 模式 B：維度並行（Stage 3-4）
```
租約1-25 順序處理，但每份內部並行
  Stage 1-2: 8分（順序）
  Stage 3: 8分（並行 2.5x）← 原 20分
  Stage 4: 5分（並行 2.2x）← 原 11分
  Stage 5: 2分（順序）

單份耗時 = 8 + 8 + 5 + 2 = 23分
總耗時 = 25 × 23分 = 575 分鐘
       = 9.6 小時
       = 1.2 個工作天

加速比 = 17.1 ÷ 9.6 = 1.78x
節省 = 7.5 小時
```

#### 模式 C：批量並行（3 Subagent）
```
第1批（租約1-3）：41分 ┐
第2批（租約4-6）：41分 ├─ 並行執行
第3批（租約7-9）：41分 ┘
...
第9批（租約25）：41分

總耗時 = 9批 × 41分 = 369 分鐘
       = 6.2 小時
       = 0.8 個工作天

加速比 = 17.1 ÷ 6.2 = 2.76x
節省 = 10.9 小時
```

#### 模式 D：雙層並行（批量 + 維度）⚡
```
第1批（租約1-3，每份內部並行）：23分 ┐
第2批（租約4-6，每份內部並行）：23分 ├─ 並行執行
第3批（租約7-9，每份內部並行）：23分 ┘
...
第9批（租約25，內部並行）：23分

總耗時 = 9批 × 23分 = 207 分鐘
       = 3.5 小時
       = 0.4 個工作天

加速比 = 17.1 ÷ 3.5 = 4.9x
節省 = 13.6 小時
```

### 成本分析

| 模式 | 耗時 | Token 消耗 | 人工等待 | 加速比 |
|------|------|-----------|---------|--------|
| A: 完全順序 | 17.1 小時 | 100% | 17.1 小時 | 1x（基準） |
| B: 維度並行 | 9.6 小時 | 102% | 9.6 小時 | 1.78x |
| C: 批量並行 | 6.2 小時 | 105% | 6.2 小時 | 2.76x |
| D: 雙層並行 | 3.5 小時 | 108% | 3.5 小時 | **4.9x** ⚡ |

**關鍵發現**：
- ✅ Token 成本僅增加 8%，但時間節省 79%
- ✅ 人工等待時間從 2 天降到半天
- ✅ 雙層並行是最優策略（加速 vs 成本最佳平衡）

## 📊 案例2：單份租約深度檢查

### 場景描述
- **任務**：單份 15 頁租約的完整檢查
- **維度**：10 個風險維度（電費、押金、修繕...）
- **品質要求**：引用準確率 > 95%

### 執行模式對比

#### 模式 A：順序執行（基準線）
```
Stage 1: OCR（5分）
  ↓
Stage 2: 切分（3分）
  ↓
Stage 3: 規則比對
  維度1: 電費（2分） → 維度2: 押金（2分） → ... → 維度10（2分）
  小計：20分
  ↓
Stage 4: 風險生成
  組件1: 風險清單（5分） → 組件2: 缺失條款（3分） → 組件3: 轉介資源（1分）
  小計：9分
  ↓
Stage 5: 信心評分（2分）

總耗時 = 5 + 3 + 20 + 9 + 2 = 39分
```

#### 模式 B：維度並行（僅 Stage 3）
```
Stage 1: OCR（5分）
  ↓
Stage 2: 切分（3分）
  ↓
Stage 3: 規則比對（並行）
  Subagent 1 → [電費 + 押金 + 修繕 + 終止權]（8分）┐
  Subagent 2 → [稅費 + 次承租 + 改建]（6分）       ├─ 並行
  Subagent 3 → [通知 + 遺失 + 其他]（4分）        ┘

  實際耗時 = max(8, 6, 4) = 8分（最長組耗時）
  ↓
Stage 4: 風險生成（順序，9分）
  ↓
Stage 5: 信心評分（2分）

總耗時 = 5 + 3 + 8 + 9 + 2 = 27分
加速比 = 39 ÷ 27 = 1.44x
節省 = 12分
```

#### 模式 C：組件並行（僅 Stage 4）
```
Stage 1-3: 順序執行（28分）
  ↓
Stage 4: 風險生成（並行）
  Subagent 1 → 風險清單（5分）┐
  Subagent 2 → 缺失條款（3分）├─ 並行
  Subagent 3 → 轉介資源（1分）┘

  實際耗時 = max(5, 3, 1) = 5分
  ↓
Stage 5: 信心評分（2分）

總耗時 = 28 + 5 + 2 = 35分
加速比 = 39 ÷ 35 = 1.11x
節省 = 4分
```

#### 模式 D：雙階段並行（Stage 3 + 4）⚡
```
Stage 1-2: 順序執行（8分）
  ↓
Stage 3: 規則比對（並行，8分）
  ↓
Stage 4: 風險生成（並行，5分）
  ↓
Stage 5: 信心評分（2分）

總耗時 = 8 + 8 + 5 + 2 = 23分
加速比 = 39 ÷ 23 = 1.70x
節省 = 16分
```

### 效率矩陣

| 模式 | Stage 3 | Stage 4 | 總耗時 | 加速比 | Token 增量 |
|------|---------|---------|--------|--------|-----------|
| A: 完全順序 | 20分 | 9分 | 39分 | 1x | 0% |
| B: 維度並行 | 8分 | 9分 | 27分 | 1.44x | +3% |
| C: 組件並行 | 20分 | 5分 | 35分 | 1.11x | +2% |
| D: 雙階段並行 | 8分 | 5分 | 23分 | **1.70x** ⚡ | +5% |

**關鍵發現**：
- ✅ Stage 3（規則比對）並行效果最顯著（12 分 → 8 分）
- ✅ Stage 4（風險生成）並行效果中等（9 分 → 5 分）
- ✅ 雙階段並行是最優策略（總計節省 16 分鐘）

## 📊 案例3：大規模批量檢查（50 份租約）

### 場景描述
- **任務**：社福機構需檢查 50 份弱勢租客的租約
- **時間限制**：1 個工作天內完成
- **品質要求**：不可降低檢查品質

### 執行模式對比

#### 模式 A：順序執行（不可行）
```
50 × 41分 = 2,050 分鐘 = 34.2 小時 = 4.3 工作天

結論：❌ 超過時間限制，無法在 1 天內完成
```

#### 模式 B：批量並行（3 Subagent）
```
17批（每批 3 份）× 41分 = 697 分鐘 = 11.6 小時

結論：❌ 勉強可行，但需要連續工作 11.6 小時
```

#### 模式 C：雙層並行（3 Subagent + 維度並行）⚡
```
17批（每批 3 份，內部並行）× 23分 = 391 分鐘 = 6.5 小時

結論：✅ 可在 1 個工作天內完成（留 1.5 小時餘量）
```

#### 模式 D：激進並行（5 Subagent + 維度並行）
```
10批（每批 5 份，內部並行）× 23分 = 230 分鐘 = 3.8 小時

結論：✅ 半天完成，但資源需求高
```

### 資源需求分析

| 模式 | 耗時 | Subagent 數 | 記憶體需求 | Token 消耗 | 可行性 |
|------|------|------------|----------|-----------|--------|
| A: 順序 | 34.2 小時 | 1 | 2GB | 100% | ❌（超時） |
| B: 批量並行 | 11.6 小時 | 3 | 6GB | 105% | 🟡（勉強） |
| C: 雙層並行 | 6.5 小時 | 3 | 6GB | 108% | ✅（最優） |
| D: 激進並行 | 3.8 小時 | 5 | 10GB | 112% | ✅（資源充足時） |

**關鍵發現**：
- ✅ 雙層並行使「不可能」變成「可行」
- ✅ 記憶體是瓶頸（每 Subagent 約 2GB）
- ✅ Token 成本增加可控（8-12%）

## 📊 案例4：緊急單件查詢（用戶場景）

### 場景描述
- **任務**：用戶上傳 1 份租約，需要立即結果
- **時間限制**：3 分鐘內給出初步結果
- **品質要求**：至少標記高風險條款

### 執行模式對比

#### 模式 A：完整檢查（順序）
```
Stage 1-5 完整執行（順序）= 41分

結論：❌ 超過用戶耐心閾值（3 分鐘）
```

#### 模式 B：快速模式（跳過低優先級）
```
Stage 1: OCR（5分）
Stage 2: 切分（3分）
Stage 3: 僅檢查 P0 高風險維度（8分）
Stage 4: 簡化報告（3分）
Stage 5: 跳過

總耗時 = 5 + 3 + 8 + 3 = 19分

結論：🟡 勉強可行，但仍超過 3 分鐘
```

#### 模式 C：快速模式 + 並行
```
Stage 1-2: 順序（8分）
Stage 3: P0 維度並行（3分）← 原 8分
Stage 4: 簡化報告（3分）

總耗時 = 8 + 3 + 3 = 14分

結論：🟡 勉強可行，但仍超過 3 分鐘
```

#### 模式 D：分階段輸出（推薦）⚡
```
第1階段（立即）：
  Stage 1-2（8分）+ Stage 3 部分結果（2分）
  → 輸出：「正在檢查，已發現 3 條高風險」

第2階段（背景執行）：
  Stage 3 完整檢查（並行，再 6分）
  Stage 4-5（7分）
  → 輸出：「檢查完成，共 8 條風險」

總耗時 = 8 + 2 + 6 + 7 = 23分
首次反饋 = 10分（用戶體感 < 15分）

結論：✅ 最優策略（兼顧速度與品質）
```

### 用戶體驗對比

| 模式 | 首次反饋 | 完整結果 | 用戶滿意度 | 技術複雜度 |
|------|---------|---------|-----------|-----------|
| A: 完整順序 | 41分 | 41分 | 低（等太久） | 低 |
| B: 快速順序 | 19分 | 19分 | 中（品質降低） | 低 |
| C: 快速並行 | 14分 | 14分 | 中（品質降低） | 中 |
| D: 分階段並行 | 10分 | 23分 | **高**（快速 + 完整） ⚡ | 高 |

**關鍵發現**：
- ✅ 分階段輸出 + 並行 = 最佳用戶體驗
- ✅ 首次反饋 10 分內可接受
- ✅ 背景執行避免用戶焦慮等待

## 🎯 最佳實踐建議

### 根據場景選擇模式

| 場景 | 推薦模式 | 理由 |
|------|---------|------|
| **單份租約（< 3 份）** | 雙階段並行（Stage 3+4） | 平衡速度與成本 |
| **小批量（3-10 份）** | 批量並行（3 Subagent） | 簡單有效 |
| **評測（25 份）** | 雙層並行（批量 + 維度） | 最大化加速 |
| **大規模（50+ 份）** | 雙層並行 + 動態調整 | 避免資源耗盡 |
| **緊急查詢** | 分階段輸出 + 並行 | 優化用戶體驗 |

### 避免過度並行

| 錯誤 | 後果 | 正確做法 |
|------|------|---------|
| **並行 OCR** | 記憶體爆炸（圖像處理） | 順序執行或小批量 |
| **並行超過 5 個 Subagent** | 資源競爭，速度反降 | 保持 3-4 個 |
| **強依賴任務並行** | 邏輯錯誤，結果錯亂 | 識別依賴關係 |
| **過度設計** | 複雜度高，維護困難 | 先簡單並行，再優化 |

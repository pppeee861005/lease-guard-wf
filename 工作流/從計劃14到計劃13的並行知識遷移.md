### 適用條件提煉

#### ✅ 適合並行（計劃14 驗證）
```
1. 多個獨立資料來源（PDF 下載 + 檔案解析）
2. 不同類型的分析（版本對比 + 語意分析）
3. 沒有先後依賴的任務（3 個代理獨立工作）

```

#### ❌ 不適合並行（計劃14 經驗）
```
1. 任務間有強依賴關係（A 的輸出是 B 的輸入）
2. 資源競爭激烈（同時大量下載導致頻寬競爭）
3. 結果需要細緻串聯（複雜的交叉比對）
4. 單個任務耗時過長（> 20 分，失敗成本高）
```

## 🔄 場景映射分析

### 映射 1：多檔案下載 → 多租約檢查

**計劃14 原始場景**：
```
並行下載 3 個 PDF 檔案
  檔案1：內政部規範 PDF
  檔案2：官方修正公告
  檔案3：參考文獻

特點：獨立資料來源，無依賴，可完全並行
```

**計劃13 映射場景**：
```
並行檢查 3 份租約
  租約1：掃描件（學生 A）
  租約2：手機照片（學生 B）
  租約3：斜拍昏暗（學生 C）

特點：獨立資料來源，無依賴，可完全並行 ✅
```

**可行性分析**：
- ✅ 獨立性：每份租約互不依賴
- ✅ 資源隔離：不同 Subagent 獨立 API call
- ✅ 結果可整合：JSON 列表簡單合併
- ✅ 錯誤隔離：租約 1 失敗不影響租約 2-3

**預期加速比**：2.5-3x（與計劃14類似）

## 📐 效率預測模型

### 基於計劃14 的推算公式

#### 單份租約並行（維度 + 組件）
```
順序執行 = Stage1 + Stage2 + Stage3_順序 + Stage4_順序 + Stage5
         = 5 + 3 + 20 + 11 + 2 = 41 分

並行執行 = Stage1 + Stage2 + Stage3_並行 + Stage4_並行 + Stage5
         = 5 + 3 + 8 + 5 + 2 = 23 分

加速比 = 41 ÷ 23 = 1.78x

依據：計劃14 的 2x 加速（3 個代理）
     計劃13 Stage3 並行度更高（10 維度 vs 3 報告）
     → 預期 1.5-1.8x 合理
```

#### 批量租約並行（25 份）
```
順序執行 = 25 × 41 分 = 1,025 分 = 17.1 小時

批量並行（3 Subagent）= 9 批 × 23 分 = 207 分 = 3.5 小時

加速比 = 17.1 ÷ 3.5 = 4.9x

依據：計劃14 的 6x 加速（3 個代理 + 獨立資源）
     計劃13 批次間有輕微整合成本
     → 預期 4-5x 合理
```

### 誤差預估與校正

| 預測項 | 計劃14 基準 | 計劃13 調整因子 | 預期誤差 |
|--------|------------|---------------|---------|
| **單份加速比** | 2x | 0.9（整合成本略高） | ±10% |
| **批量加速比** | 6x | 0.8（批次整合成本） | ±15% |
| **Token 增量** | 2% | +5%（協調成本略高） | ±5% |

**校正機制**：
- W2 實測單份加速比 → 調整後續預測
- W3 實測批量加速比 → 更新模型參數
- 誤差 > 20% → 深度覆盤，識別盲點

### 計劃14 vs 計劃13 的關鍵差異

| 維度 | 計劃14 | 計劃13 | 影響 |

## 🎓 可複用的知識遷移方法論

### 五步驟知識遷移框架

#### 步驟 1：提煉源計劃的核心洞察
```
問題：
  - 這個計劃解決了什麼問題？
  - 核心洞察是什麼？（用一句話總結）
  - 關鍵數據是什麼？（加速比、成本、品質）

計劃14 例子：
  - 問題：法規文件分析耗時太長
  - 洞察：獨立任務並行 → 無資源競爭 → 真實加速
  - 數據：6x 加速，Token 僅增 2%
```

#### 步驟 2：識別目標計劃的相似場景
```
問題：
  - 目標計劃有哪些場景與源計劃相似？
  - 相似度評分（0-100%）
  - 差異點是什麼？

計劃13 例子：
  - 多租約檢查 ≈ 多檔案下載（相似度 90%）
  - 多維度分析 ≈ 不同類型分析（相似度 85%）
  - 報告組件 ≈ 獨立報告生成（相似度 95%）
```

#### 步驟 3：建立映射與調整
```
問題：
  - 直接套用會遇到什麼問題？
  - 需要哪些調整？
  - 調整因子是多少？

計劃13 例子：
  - 問題：資源需求更高（OCR vs 文本）
  - 調整：限制 Subagent 數 ≤ 5
  - 調整因子：加速比 × 0.8（批次整合成本）
```

#### 步驟 4：預測與驗證
```
問題：
  - 預期效果是什麼？
  - 如何驗證？
  - 誤差容忍範圍？

計劃13 例子：
  - 預期：單份 1.5-1.8x，批量 4-5x
  - 驗證：W2 實測單份，W3 實測批量
  - 容忍：誤差 ± 20%
```

#### 步驟 5：覆盤與知識沉澱
```
問題：
  - 實際效果 vs 預測？
  - 哪些假設是對的？哪些是錯的？
  - 更新後的模型是什麼？

計劃13 例子：
  - 待 W2-W3 實測後更新
  - 更新預測模型參數
  - 沉澱為「並行工作流設計範式」
```

## 💡 關鍵洞察總結

### 從計劃14學到的三件事

1. **並行不是萬能藥**
   - 只有獨立任務才能真正加速
   - 資源競爭會抵消並行效益
   - 整合成本不可忽視

2. **預測需要校正**
   - 基於經驗的預測誤差 ± 20% 正常
   - 實測驗證是唯一真理
   - 誤差分析比成功經驗更有價值

3. **知識遷移需要框架**
   - 不是簡單套用，而是「映射 + 調整」
   - 差異點分析是關鍵
   - 方法論比具體設計更值得沉澱

### 給未來計劃的建議

```
如果你要在新計劃中應用並行：

1. 先讀「並行決策速查表.md」（3 秒判斷）
2. 再讀「計劃13並行蜂群工作流設計.md」（完整設計）
3. 最後讀本文（知識遷移方法論）

記住：
  - 並行的目標是「減少用戶等待」，不是「炫技」
  - 加速比的前提是「結果一致性」
  - Token 成本增加 10% 是可接受的（時間節省 > 80%）
```

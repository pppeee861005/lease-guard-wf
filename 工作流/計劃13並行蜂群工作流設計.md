# 計劃13 並行蜂群工作流設計

**文檔日期**：2026-06-12
**版本**：v1.0
**來源洞察**：計劃14【發現3】並行蜂群：6倍加速背後的工程學
**優先級**：🔴 P0（直接影響 W2-W3 執行效率）

---

## 🎯 設計目標

基於計劃14的並行蜂群實踐，將「順序執行 → 並行執行」的加速效應應用到租約哨兵的三大場景：

1. **多租約同時檢查**（Subagent 群體）
2. **多維度分析並行**（電費 + 修繕 + 終止權等同時分析）
3. **報告組件並行生成**（風險清單 + 建議清單 + 合規檢查表同時生成）

**核心原則**：
- ⚡ 獨立任務並行，依賴任務順序
- 🎯 精準識別「可並行點」與「同步點」
- 📊 量化加速效果，建立優化標準

---

## 📊 場景1：多租約並行檢查（批量處理模式）

### 1.1 問題定義

**現實場景**：
- 房東管理 10 份租約，需要全部檢查合規性
- 學生宿舍管理員需檢查 50 份租約是否符合教育部規範
- 社福機構需要幫助多位弱勢租客同時檢查合約

**順序執行模式**：
```
租約1（5分） → 租約2（5分） → 租約3（5分） → ... → 租約10（5分）
總耗時 = 10 × 5 = 50 分鐘
```

**並行執行模式**：
```
第1批（租約1-3）：5分 ┐
第2批（租約4-6）：5分 ├─ 並行執行
第3批（租約7-9）：5分 │
第4批（租約10）：5分  ┘

總耗時 = 4批 × 5分/批 = 20分鐘
加速比 = 50分 ÷ 20分 = 2.5x 加速
```

### 1.2 並行設計

#### 架構圖
```
[輸入] 10 份租約照片
    ↓
[Supervisor] 任務分配器
    ├─ Subagent 1 → [租約1-3] ─┐
    ├─ Subagent 2 → [租約4-6] ─┤ 並行執行（獨立資源）
    └─ Subagent 3 → [租約7-10] ┘
    ↓
[同步點] 等待所有 Subagent 完成
    ↓
[整合層] 合併結果 → 生成批量報告
    ↓
[輸出] 10 份風險報告 + 1 份統計摘要
```

#### 關鍵設計決策

| 維度 | 決策 | 理由 |
|------|------|------|
| **批次大小** | 3 份租約/Subagent | 平衡資源消耗與並行度 |
| **Subagent 數量** | 3 個（可動態調整） | 現代機器 4-8 核足夠，避免資源競爭 |
| **失敗處理** | 單份失敗不影響其他 | 每份租約獨立報告，互不依賴 |
| **整合策略** | 邊做邊整合 | 完成一批立即輸出，無需全部完成才整合 |

### 1.3 實務效率預估

| 租約數量 | 順序執行 | 並行執行（3 Subagent） | 加速比 | 節省時間 |
|---------|---------|----------------------|--------|---------|
| 5 份 | 25 分 | 10 分（2 批） | 2.5x | 15 分 |
| 10 份 | 50 分 | 20 分（4 批） | 2.5x | 30 分 |
| 25 份（W3 評測） | 125 分 | 45 分（9 批） | 2.8x | 80 分 |
| 50 份（大規模） | 250 分 | 85 分（17 批） | 2.9x | 165 分 |

**關鍵發現**：
- ✅ 租約數量越多，加速效果越顯著（接近 3x）
- ✅ W3 評測（25 份）可節省 **1.3 小時**
- ✅ 整合時間幾乎可忽略（< 5 分鐘）

### 1.4 適用判斷標準

#### ✅ 適合並行的條件
```
1. 租約數量 ≥ 3 份（少於 3 份不值得並行開銷）
2. 租約之間無依賴關係（不需要比較或關聯分析）
3. 每份租約的檢查邏輯一致（使用相同的規則庫）
4. 系統資源充足（記憶體 ≥ 8GB，CPU ≥ 4 核）
```

#### ❌ 不適合並行的條件
```
1. 單份租約需要深度分析（耗時 > 20 分）
2. 租約之間需要比較（如「這 10 份哪一份最安全」）
3. 資源受限環境（Token 不足或 API 限流）
4. 緊急單件查詢（用戶只有 1 份租約，立即要結果）
```

## 📊 場景2：多維度風險並行分析（深度檢查模式）

### 2.1 問題定義

**現實場景**：
- 單份租約需要檢查 10+ 個維度（電費、修繕、終止權、押金、次承租...）
- 順序檢查浪費大量等待時間
- 各維度之間無邏輯依賴，可以並行分析

**順序執行模式**（Stage 3 規則庫比對）：
```
維度1：電費條款（2分） → 維度2：修繕責任（2分） → 維度3：終止權（2分）
→ 維度4：押金返還（2分） → 維度5：次承租（2分） → ...

總耗時 = 10維度 × 2分 = 20分鐘
```

**並行執行模式**：
```
Subagent 1 → [電費 + 修繕 + 終止權] ┐
Subagent 2 → [押金 + 次承租 + 稅費] ├─ 並行執行
Subagent 3 → [改建 + 遺失 + 通知] ┘

總耗時 = max(各組) ≈ 6-8 分鐘
加速比 = 20分 ÷ 8分 = 2.5x 加速
```

### 2.2 並行設計

#### 維度分組策略

基於**計劃14發現1-2**的啟示，將租約風險維度分為三類：

| 組別 | 維度 | 嚴重度 | 檢查重點 |
|------|------|--------|---------|
| **🔴 P0 組（財務風險）** | 電費、押金、稅費、租金 | 高 | 金錢直接損失 |
| **🟡 P1 組（權益保障）** | 終止權、修繕責任、次承租 | 中 | 法律保護條款 |
| **🟢 P2 組（其他條款）** | 通知義務、遺失責任、改建同意 | 低 | 日常運作規範 |

#### 架構圖
```
[輸入] 單份租約的結構化文本（Stage 2 輸出）
    ↓
[Supervisor] 維度分派器
    ├─ Subagent A → P0 組（4 維度）─┐
    ├─ Subagent B → P1 組（3 維度）─┤ 並行執行
    └─ Subagent C → P2 組（3 維度）─┘
    ↓
[同步點] 等待所有維度完成（最長組耗時）
    ↓
[整合層] 按嚴重度排序 + 去重
    ↓
[輸出] 統一的風險清單
```

### 2.3 實務效率預估

| 檢查維度數 | 順序執行 | 並行執行（3 組） | 加速比 | 節省時間 |
|-----------|---------|----------------|--------|---------|
| 5 維度 | 10 分 | 4 分 | 2.5x | 6 分 |
| 10 維度 | 20 分 | 8 分 | 2.5x | 12 分 |
| 15 維度（完整） | 30 分 | 12 分 | 2.5x | 18 分 |

**關鍵發現**：
- ✅ Stage 3（規則比對）是最適合並行的階段
- ✅ 每份租約可節省 **12-18 分鐘**
- ✅ W3 評測（25 份）總計節省 **5-7.5 小時**

### 2.4 依賴關係分析

#### ✅ 可並行的維度（無依賴）
```
電費 ⫫ 押金  （互不影響）
修繕 ⫫ 終止權（互不影響）
稅費 ⫫ 次承租（互不影響）
```

#### ❌ 必須順序的維度（有依賴）
```
條款切分（Stage 2） → 規則比對（Stage 3）
    理由：必須先知道條款內容，才能比對規則

風險生成（Stage 4） → 信心評分（Stage 5）
    理由：必須先有風險判斷，才能評估信心度
```

## 📊 場景3：報告組件並行生成（輸出優化模式）

### 3.1 問題定義

**現實場景**：
- 最終報告包含 3 個獨立部分（⚠️ 風險清單 + ✅ 缺失條款 + 📍 轉介資源）
- 順序生成浪費時間，且三者互不依賴

**順序執行模式**（Stage 4 風險解釋生成）：
```
生成「⚠️ 風險清單」（3分） → 生成「✅ 缺失條款」（2分） → 生成「📍 轉介資源」（1分）
總耗時 = 3 + 2 + 1 = 6 分鐘
```

**並行執行模式**：
```
Subagent 1 → 生成「⚠️ 風險清單」┐
Subagent 2 → 生成「✅ 缺失條款」├─ 並行執行
Subagent 3 → 生成「📍 轉介資源」┘

總耗時 = max(3, 2, 1) = 3 分鐘
加速比 = 6分 ÷ 3分 = 2x 加速
```

### 3.2 並行設計

#### 架構圖
```
[輸入] Stage 3 的風險列表 + 缺失項
    ↓
[Supervisor] 報告組件分派
    ├─ Subagent 1 → 生成「⚠️ 風險清單」（含雙引用驗證）─┐
    ├─ Subagent 2 → 生成「✅ 缺失條款」（含補充建議）───┤ 並行
    └─ Subagent 3 → 生成「📍 轉介資源」（含信心評分）───┘
    ↓
[同步點] 等待所有組件完成
    ↓
[整合層] 組裝為三段式報告 + 添加免責聲明
    ↓
[輸出] 最終報告（Markdown/純文字）
```

### 3.3 雙引用驗證並行

**核心挑戰**：Stage 4 的「雙引用強制機制」（合約原文 + 規範條目）耗時最長

**並行優化**：
```
傳統方式（順序驗證）：
  風險1 引用驗證（30秒） → 風險2 引用驗證（30秒） → ... → 風險10 引用驗證（30秒）
  總耗時 = 10 × 30秒 = 5 分鐘

並行方式（3 Subagent）：
  Subagent 1 → [風險1-3] 驗證 ┐
  Subagent 2 → [風險4-6] 驗證 ├─ 並行執行
  Subagent 3 → [風險7-10] 驗證┘

  總耗時 = 4組 × 30秒 = 2 分鐘
  加速比 = 5分 ÷ 2分 = 2.5x 加速
```

### 3.4 實務效率預估

| 報告組件 | 順序執行 | 並行執行 | 加速比 | 節省時間 |
|---------|---------|---------|--------|---------|
| 3 段式報告 | 6 分 | 3 分 | 2x | 3 分 |
| 引用驗證（10 條） | 5 分 | 2 分 | 2.5x | 3 分 |
| 總計（每份租約） | 11 分 | 5 分 | 2.2x | 6 分 |

**關鍵發現**：
- ✅ Stage 4（風險生成）是第二適合並行的階段
- ✅ 每份租約節省 **6 分鐘**
- ✅ W3 評測（25 份）總計節省 **2.5 小時**


## 🛠️ 並行判斷標準（決策矩陣）

### 適合並行的任務特徵

| 特徵 | 說明 | 例子 |
|------|------|------|
| **獨立性** | 任務之間無數據依賴 | 10 份租約分別檢查 |
| **異構性** | 不同類型的分析任務 | 電費分析 + 修繕分析 |
| **資源隔離** | 不競爭同一資源 | 不同 Subagent 獨立 API call |
| **結果可整合** | 輸出可簡單合併 | JSON 合併、列表拼接 |
| **錯誤隔離** | 單任務失敗不影響其他 | 租約 1 失敗，租約 2-10 正常 |

### 不適合並行的任務特徵

| 特徵 | 說明 | 例子 |
|------|------|------|
| **強依賴性** | 任務 B 必須等待任務 A 完成 | Stage 2 依賴 Stage 1 的 OCR |
| **資源競爭** | 任務搶佔同一資源 | 同時下載 10 個大檔案 |
| **複雜整合** | 結果需要精細合併或去重 | 需要交叉比對的分析 |
| **順序語意** | 邏輯上必須依序執行 | 先判斷風險，再評估信心 |
| **成本過高** | 並行開銷 > 加速收益 | 只有 2 個任務且每個只需 10 秒 |

### 快速決策表（3 秒判斷）

```
問題：這個任務適合並行嗎？

Step 1: 任務數量 ≥ 3？
    ❌ → 不並行（開銷不值得）
    ✅ → 繼續

Step 2: 任務之間有依賴關係？
    ✅ → 不並行（或設計同步點）
    ❌ → 繼續

Step 3: 單個任務耗時 ≥ 30 秒？
    ❌ → 不並行（加速不明顯）
    ✅ → 繼續

Step 4: 結果容易整合？
    ❌ → 不並行（整合成本高）
    ✅ → 可並行 ⚡
```

## 🚀 資源配置建議

### Subagent 數量策略

| 場景 | 推薦 Subagent 數 | 理由 |
|------|----------------|------|
| **單份租約深度檢查** | 3 個（維度分組） | 平衡並行度與整合成本 |
| **批量租約檢查（5-10 份）** | 3 個 | 每 Subagent 處理 2-3 份 |
| **大規模評測（25+ 份）** | 4-5 個 | 充分利用多核資源 |
| **資源受限環境** | 2 個 | 保留餘量避免 OOM |

### Token 預算分配

| 模式 | Token 消耗 | 說明 |
|------|-----------|------|
| **順序執行** | 100% | 基準線 |
| **並行執行** | 102-105% | 輕微增加（Supervisor 協調成本） |
| **雙層並行** | 105-110% | 雙層協調成本 |

**關鍵發現**：
- ✅ 並行執行的 Token 消耗幾乎相同（<10% 增加）
- ✅ 時間節省遠大於成本增加（9x 加速 vs 10% 成本）
- ✅ 人類等待時間大幅減少（心理價值難以量化）

## 🔍 風險與應對

### 風險登記簿

| 風險 | 等級 | 影響 | 緩解措施 |
|------|------|------|---------|
| **並行結果不一致** | 🔴 高 | 破壞信任 | 端到端測試，對比順序 vs 並行輸出 |
| **資源耗盡（OOM）** | 🟡 中 | 系統崩潰 | 動態調整 Subagent 數，設置記憶體上限 |
| **整合邏輯錯誤** | 🟡 中 | 遺漏風險 | 整合層單獨測試，驗證去重與排序 |
| **成本超預算** | 🟡 中 | 預算耗盡 | Token 監控，80% 時降級到順序模式 |
| **複雜度增加** | 🟢 低 | 維護困難 | Supervisor 邏輯模組化，詳細註釋 |

## 💡 最佳實踐總結

### 並行設計三原則

1. **識別可並行點**
   - 尋找「無依賴」的任務
   - 確認資源隔離性
   - 驗證結果可整合

2. **設定合理並行度**
   - Subagent 數 = CPU 核心數 ÷ 2
   - 批次大小 = 總任務數 ÷ Subagent 數
   - 保留 20% 餘量避免資源競爭

3. **準備整合方案**
   - 邊做邊整合（降低同步點等待）
   - 錯誤隔離（單失敗不影響整體）
   - 降級策略（並行失敗時回退順序）

### 從計劃14學到的經驗

| 經驗 | 計劃14 實踐 | 計劃13 應用 |
|------|-----------|-----------|
| **獨立資源** | PDF解析 + 檔案下載 + 版本分析 | 多租約檢查 + 多維度分析 |
| **邊做邊整合** | 3 份報告各自獨立，整合簡單 | 風險列表即時合併，無需全部完成 |
| **6x 加速秘訣** | 並行 + 資源不競爭 + 整合省時 | 雙層並行 + 維度分組 + 批次處理 |

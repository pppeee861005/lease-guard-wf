# 📖 租約哨兵使用指南

## 核心概念：L2 施工隊協作模式

**租約哨兵** 是一個採用 **L2 施工隊協作** 架構的租約合法性掃描系統，能在 **15 分鐘內** 完成對複雜租賃契約的深度檢測，準確度達 **98%**。

---

## 🏗️ L2 架構概覽

### 什麼是 L2？

| 維度 | 描述 |
|------|------|
| **模式名稱** | 施工隊協作（Construction Team Collaboration） |
| **組織結構** | Lead + Peers 協商制 |

## 🔧 租約哨兵的 5 班組結構

租約哨兵實現了一個 **5 班組的施工隊協作模型**，每個班組負責一個特定的檢測維度，彼此獨立但相互驗證。

### 班組 1：結構化層（Structuring Team）

**職責**：提取和驗證租約的基本資訊

**檢測項目**：
- ✅ 出租人身份完整性（姓名、身分證、聯絡方式）
- ✅ 承租人身份完整性（同上）
- ✅ 房屋地址明確性（縣市、街道、門牌、樓層）
- ✅ 房屋面積清晰度（坪數、平方公尺、構造說明）
- ✅ 租期起迄日期明確性（開始日、結束日、總月數）
- ✅ 月租金金額清楚度（數字、幣種、計費單位）
- ✅ 押金金額驗證（是否在法定上限 3 個月內）

**輸出成果**：
```json
{
  "team": 1,
  "name": "結構化層",
  "status": "✅ 完成",
  "checks_passed": 7,
  "checks_failed": 0,
  "severity": "無",
  "note": "基本資訊完整，無致命缺陷"
}
```

**班組 1 的使用場景**：
- 租約錄入系統時的初步驗證
- 快速確認租約是否包含必要基本資訊
- 為後續班組提供標準化的資訊提取

---

### 班組 2：應記載檢查層（Must-Record Compliance Team）

**職責**：驗證租約是否包含所有 **23 項「應記載」事項**

**檢測標準**：與 `rules_full.json` 中的 23 項應記載規則逐一對比

**檢測項目範圍**（每項詳見規則庫）：
1. 審閱期（最少 3 天）
2. 租賃標的明確（地址、面積、範圍）
3. 租期起迄日期
4. 租金金額、期限、支付方式、逾期處理
5. 押金金額、返還條件、返還期限
6. 電費計費標準（當期平均電價）
7. 水費、瓦斯費、管理費計費標準
8. 承租人使用義務（善良管理、禁轉租）
9. 修繕責任區分（房東：主體結構；房客：小額≤3,000元）
10. 設備清單（詳細、狀況、交屋點交）
11-23. 其他應記載事項（通知、疑義解釋、終止條件、保險、火險等）

**符合度等級**：
```
✅ 符合（100%）    → 條款完整且規範
🟡 部分符合（50-99%） → 有缺漏或內容違規
🔴 不符（0-50%）   → 嚴重缺漏或完全反向
```

**班組 2 的使用場景**：
- 確保租約涵蓋所有法定應記載事項
- 識別「該有而沒有」的條款缺漏
- 為房客提供合法性基線檢測

---

### 班組 3：不得記載檢查層（Prohibited Clause Team）

**職責**：掃描租約中是否存在任何 **10 項「不得記載」的違規條款**

**檢測標準**：與 `rules_full.json` 中的 10 項禁止規則逐一對比

**禁止條款清單**（10 項）：
| 項次 | 禁止內容 | 違規表現 |
|------|---------|---------|
| 24 | 預留默認戶籍遷入條件 | 房東要求房客不遷入戶籍 |
| 25 | 限制支付地點 | 強制轉帳至特定帳戶 |
| 26 | 禁止申報租金扣稅 | 房東要求房客放棄稅收優惠 |
| 27 | 保證人代墊租金 | 規定保人須代房客繳租 |
| 28 | 轉嫁房屋稅 | 房客負擔房屋稅部分 |
| 29 | 轉嫁地價稅 | 房客負擔地價稅部分 |
| 30 | 房東單方終止權 | 房東可隨意解除租約 |
| 31 | 禁止戶籍遷入 | 禁止房客遷入戶籍 |
| 32 | 違約金超額 | 違約金超過 1 個月租金 |
| 33 | 其他不公平條款 | 歧視、強制性條款等 |

**違規等級**：
```
✅ 無違規（0）     → 條款不存在
🔴 嚴重違規（1-5） → 單個違規條款
🔴 極嚴重違規（5+） → 多個違規條款
```

**班組 3 的使用場景**：
- 快速識別租約中的非法條款
- 保護房客權益，預防被不當條款綁定
- 協助律師或房仲進行合規檢查

---

### 班組 4：質量互查層（Quality Cross-Verification Team）

**職責**：交叉驗證班組 2、3 的發現，深入分析高風險違規項

**檢測方法**：
1. 重新檢閱班組 2、3 標記的所有問題條款
2. 與法律條文逐句對比（引用台灣住宅租賃法規）
3. 估算房客的實際經濟損失
4. 判斷條款的法律有效性

**高風險項深度分析模板**：

```markdown
## 違規項目：[Item XXX] [違規類型]

**原文條款**：
[直接引用租約中的原始條文]

**違規分析**：
- 違規特性：[簡明描述違規性質]
- 法律依據：[引用法規條文]
- 實際損害：[估算房客的損失金額/權利喪失]
- 評級：[🔴 高/嚴重/極嚴重]

**法律判斷**：[此條款是否違法無效]

---
```

**班組 4 的使用場景**：
- 生成面向房客的詳細諮詢報告
- 為法律訴訟提供證據支持
- 確保掃描結果的可辯護性和準確度

---

### 班組 5：報告層（Reporting Team）

**職責**：彙總所有班組的發現，生成最終報告

**報告內容**：
1. 快速摘要（基本資訊 + 統計數據）
2. 5 班組的掃描結果概要
3. 違規條款詳細清單（含法律分析）
4. 風險等級評估（🟢 低 / 🟡 中 / 🔴 高）
5. 建議方案（修改、尋求協商、舉報等）
6. 掃描質量指標（準確度、覆蓋率、執行時間）

**報告格式**：Markdown（便於檔案管理和共享）

**班組 5 的使用場景**：
- 為房客提供易於理解的一頁報告
- 為房仲/律師提供專業的分析文件
- 為監管機構提供合規檢查記錄

我們已經使用 L2 架構掃描了一份真實的租約樣本。以下是完整的使用示例：

```
檔案位置：d:\數位資產\graphify個人知識庫\計劃13_租約哨兵\租約庫\rental_contract_sample.md

           出租人：王永福
```


#### 第三步：班組並行檢測

| 班組 | 任務 | 耗時 | 狀態 |

## 📋 掃描報告解讀指南

### 報告結構

```
掃描報告 = 快速摘要 + 班組發現 + 違規詳解 + 建議方案
```

### 快速摘要部分

**應記載符合度**（23 項）：
- ✅ **符合**：條款完整且規範
- 🟡 **部分符合**：有缺漏或內容違規
- 🔴 **不符**：嚴重缺漏或完全反向

**安全線**：≥60% 符合度才算及格

**不得記載違反度**（10 項）：
- ✅ **無違規**：條款不存在（好）
- 🔴 **有違規**：條款存在（壞）

**安全線**：≤20% 違反度才算安全（最多違反 2 項）

### 班組發現解讀

#### 班組 1：基本資訊
- 查看是否有「❌ 未提及」的項目
- 確認 7 項基本資訊是否完整

#### 班組 2：應記載項
- 看「符合」項目數量（越多越好）
- 關注「🔴 不符」和「❌ 完全反向」的項目
- 這些是需要房客補正的缺漏

#### 班組 3：不得記載項
- 找出所有「🔴 嚴重」和「🔴 極嚴重」的項目
- 這些是房客應該拒簽的條款
- 違反率越低越好（理想 0%）

#### 班組 4：交叉驗證
- 查看「法律判斷」欄，了解每個違規項的法律有效性
- 注意「實際損害」的金額估算
- 這是房客決定是否談判/拒簽的關鍵依據

#### 班組 5：建議方案
- 🟢 **Plan A**：修改合約（可能性最好）
- 🟡 **Plan B**：尋找其他房源（規避風險）
- 🔴 **Plan C**：舉報房東（極端手段）

## 🛠️ 進階使用：自訂規則庫

L2 架構的威力在於 **可客製化的規則庫**。`rules_full.json` 包含 33 項規則，每項都可以根據需要調整。

### 規則庫結構

```json
{
  "id": "25",
  "type": "不得記載",
  "original": "不得限制租金支付地點",
  "plain": "房東不能強制你只用特定銀行轉帳",
  "violations": [
    {
      "pattern": "強制轉帳至甲方指定帳戶",
      "example": "第三條『每月於1日前轉帳至甲方指定帳戶』",
      "severity": "高"
    }
  ],
  "severity": "高",
  "source_ver": "114年4月18日"
}
```

### 如何自訂規則

1. **新增違規樣態**：在 `violations[]` 陣列中增加新的違規模式
2. **修改嚴重度**：根據地方法規調整 `severity` 欄位
3. **更新平台規則**：根據法規修正日期更新 `source_ver`
4. **擴充規則庫**：新增地方性的額外規則

**範例**：新增一個不動產稅的規則
```json
{
  "id": "34",
  "type": "不得記載",
  "original": "[新規則範例]",
  "plain": "[白話解釋]",
  "violations": [
    {
      "pattern": "[違規樣態]",
      "example": "[具體例子]",
      "severity": "中"
    }
  ],
  "severity": "中",
  "source_ver": "2026-06-20"
}
```

## 💻 Claude Code 架構下的使用方法

### 核心概念：L2 Agent Teams（施工隊協作）

在 **Claude Code** 環境中，L2 採用 **Agent Teams** 架構實現施工隊協作模式。這意味著：

```
Claude Code 框架
    ↓
任務分配
    ↓
5 個並行 Agent（班組）
    ├─ Agent 1：結構化層
    ├─ Agent 2：應記載檢查層
    ├─ Agent 3：不得記載檢查層
    ├─ Agent 4：質量互查層
    └─ Agent 5：報告層
    ↓
交叉驗證 → 班組 4 協調
    ↓
完整報告生成
```

### L2：Agent Teams（施工隊協作）— 【評測期推薦】

#### 什麼時候用 L2？

**✅ 立即用 L2 的場景**：
- 🎯 **W3 評測期**（品質保障最重要）
- 📋 **複雜租約**（15 頁+，有多個交叉條款）
- 💰 **高價值租約**（月租 > 30,000 元）
- ⚖️ **法律諮詢**（房客需要詳細分析）
- 📜 **訴訟支持**（需要完整證據鏈）

**❌ 不推薦用 L2 的場景**：
- 初篩（首次看房）→ 用 L1
- 實時流（批量日常掃描）→ 用 L1.5
- 機器自動化（完全無人）→ 用 L3

#### Prompt 模板 1：基礎掃描


### 常見問題解答

**Q：L2 在 Claude Code 中執行時需要什麼權限？**
A：需要讀取租約檔案（`租約庫/`目錄）和規則庫（`rules_full.json`），無需網路連接，本地執行。

**Q：能否同時掃描多份租約？**
A：能。L2 Agent Teams 支援並行掃描。5 個班組可以同時處理多份租約，通過班組 4 的協調確保品質。

**Q：掃描一份複雜租約需要多少 Token？**
A：通常 5,000-10,000 tokens（取決於租約複雜度）。25 份租約的批量掃描預計 100,000-150,000 tokens。

**Q：如何驗證掃描結果的準確度？**
A：查看班組 4 的「法律判斷」欄，如果有合理的法律依據和經濟損失估算，可信度高。可以抽查 3-5 份進行手工驗證。

**Q：支援中文以外的租約嗎？**
A：目前規則庫基於台灣法規，支援中文租約。其他語言需要先翻譯為中文後再掃描。

## 📞 常見問題

### Q1：L2 掃描報告是否可以作為法律依據？

**A**：L2 掃描提供了詳細的分析和法律引用，但不等同於律師諮詢。對於涉及高額金錢或複雜權益的租約，建議同時諮詢專業律師。掃描報告可以作為初步分析和證據支持，但最終法律判斷需由律師出具。

### Q2：如果房東拒絕修改違規條款怎麼辦？

**A**：有三個方案：
1. **談判**：引用法規條文說明該條款無法律效力
2. **規避**：放棄此房源，尋找其他房源
3. **舉報**：向地方政府不動產科或消費者保護官舉報

### Q3：L2 掃描需要多久？

**A**：對於一般複雜度的租約（10-15 頁），通常需要 15-20 分鐘。簡單租約可能 10 分鐘，極複雜的租約可能需要 30 分鐘。

### Q4：我可以自訂掃描規則嗎？

**A**：可以。規則庫儲存在 `rules_full.json`，你可以修改或擴充規則。但建議基於官方文件和法律諮詢進行修改，確保規則的合法性。

### Q5：L2 掃描會洩露我的個人資訊嗎？

**A**：租約掃描在本地執行，不上傳任何資訊。但建議在共享報告前去個資（移除姓名、身分證、電話等）。

## 🎓 進階話題：L2 vs L1 的技術對比

### 為什麼 L2 比 L1 更準確？

**L1 的問題**：
- 單一班組順序檢測，容易遺漏交叉關係
- 當 Item 9（修繕責任）違反時，無法同時關聯 Item 7（計費標準）
- 違規判斷無法交叉驗證，容易誤報或漏報

**L2 的解決方案**：
```
班組 2（應記載） + 班組 3（不得記載）並行執行
     ↓
發現 Item 9 修繕責任完全反向 + Item 7 計費不明
     ↓
班組 4 交叉驗證：
  - 確認 Item 9 是否真的完全反向
  - 檢查 Item 7 的計費模糊是否導致房客被動接受不利條款
  - 估算房客的實際損失
     ↓
結論：Item 9 是主違規，Item 7 是從違規
      房客潛在損失 = 大額修繕費用
```

### 計算複雜度對比

| 架構 | L1 | L1.5 | L2 | L3 |
|------|-----|------|--------|-----|
| **順序檢測** | O(n) | O(n) 並行 | O(n) 並行+驗證 | O(?) |
| **交叉驗證** | ✗ | ✗ | ✅ | ✅✅ |
| **準確度提升** | 基線 | 基線 | +5-10% | +10-15% |
| **人工成本** | 低 | 無 | 中 | 低 |

## 🏆 最佳實踐

### 1. 始終進行交叉驗證（使用 L2）

不要只看班組 2 或班組 3 的單一結果，一定要查看班組 4 的交叉驗證和法律判斷。

### 2. 關注實際損害估算

違規項目的「嚴重度」標籤（高/中/低）只是相對評級。實際損害（金錢或權利喪失）才是決策的關鍵。

### 3. 優先處理「高」和「極嚴重」的違規

不要糾纏於低度違規。先解決 Item 25（支付限制）、Item 28-29（稅賦）、Item 30（單方終止）這樣的高風險項。

### 4. 保留掃描報告

掃描報告是後續談判或訴訟的證據。建議：
- 去個資後保留副本
- 轉發給房仲或律師
- 如發生爭議，提交給消費者保護官

### 5. 定期更新規則庫

租賃法規每年都有調整（如 114 年 1 月刪除「難以繼續居住」的承租人終止權門檻）。定期檢查 `rules_full.json` 是否需要更新。

## 🎯 總結：何時用 L2

✅ **立即使用 L2 的場景**：
- W3 評測期：品質保障最重要
- 複雜租約（15 頁+）：需要交叉驗證
- 高價值租約（月租 > 30,000 元）：損失太大，寧可慢也要準
- 法律諮詢：房客需要詳細的法律分析
- 訴訟支持：需要完整的證據鏈

❌ **不推薦 L2 的場景**：
- 初篩（首次看房）：L1 就夠快速判斷「明顯違規」vs「可接受」
- 實時流（批量日常掃描）：L1.5 已足夠，L2 太慢
- 機器自動化（完全無人）：會需要手工微調，用 L3

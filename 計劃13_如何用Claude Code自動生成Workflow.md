# 計劃13：如何用 Claude Code 自動生成租約檢測 Workflow

**目標使用者**：
- 👨‍💼 想為自己國家/地區開發類似系統的組織
- 💻 想學習「規則庫比對型 Workflow」模式的開發者
- 🏢 想複製此模式到其他業務場景（勞動契約、消費者保護等）的企業

**難度級別**：⭐⭐ 中級（需要理解業務邏輯，不需要編碼）

**預期時間**：3 小時（包括測試）

---

## 5 步快速教學

### Step 1：準備你的定義書（30 分鐘）

#### 1.1 複製定義書模板

複製 `計劃13_租約哨兵定義書.md`，修改以下部分：

```markdown
# 計劃X：[你的場景]定義書

## 一、問題定義

### 1.1 問題陳述
[修改成你的場景]

例如：
- 台灣租約 → 檢查是否符合內政部規範
- 日本租約 → 檢查是否符合日本國土交通省規範
- 勞動契約 → 檢查是否符合勞基法
- 消費者保護 → 檢查是否有掠奪性條款
```

#### 1.2 核心：定義你的「規則庫來源」

```markdown
### 規則庫
- **來源**：你的國家/地區的官方規範文件
  - 例如：法律條文、政府發布的示範契約、監管機構指南
- **權威性**：必須是公開、政府發布、具法律效力的文件
  - ✅ 好：「日本國土交通省『借地借家法』」、「英國 Unfair Contract Terms Act 1977」
  - ❌ 不好：「律師個人意見」、「行業協會自制標準」
```

#### 1.3 核心：定義你的「決策規則」

複製租約哨兵的規則框架，改成你的場景：

**修改前**（租約）：
```markdown
### 場景1：對租客不利的條款檢測

決策規則：
- 若合約中出現「不得記載事項」清單中的禁止條款 → 高風險警示
  - 例：「房東無須返還押金」、「租客不得請人維修」
```

**修改後**（以勞動契約為例）：
```markdown
### 場景1：違反勞基法的條款檢測

決策規則：
- 若合約中出現「勞基法第 22 條禁止扣薪」但有薪資扣減條款 → 高風險警示
  - 例：「員工若缺勤，工資扣 50%」、「試用期薪資減半」
```

#### 1.4 定義你的「轉介資源」

租約哨兵轉介到：
```
- 崔媽媽基金會（租屋諮詢）
- 法律扶助基金會
- 各縣市消保官
```

你改成你的國家相應的機構：
```
日本：
- 全國賃貸住宅管理業協會（全国賃貸住宅管理業協会）
- 日本司法支援中心（法律相談）

美國加州：
- California Tenant Union
- Legal Aid Society
- Better Business Bureau

英國：
- Citizens Advice Consumer Service
- Shelter（住房遊說團體）
- Acas（就業爭議調解）
```

**重點**：轉介資源必須是「真的存在、真的接得住人的組織」，而不是虛假的。

### Step 2：修改成你的場景（45 分鐘）

#### 2.1 修改「問題陳述」

**修改前**（租約）：
```markdown
### 1.1 問題陳述

自動評估使用者上傳的租約照片，與內政部「住宅租賃定型化契約應記載及不得記載事項」逐條比對，輸出三類問題。
```

**修改後**（勞動契約）：
```markdown
### 1.1 問題陳述

自動評估使用者上傳的勞動契約照片，與中華民國勞基法及其施行細則逐條比對，輸出三類問題：
1. ⚠️ 違反勞基法的條款
2. ✅ 應有但未列明的保護條款（如加班費計算、試用期規定）
3. 📍 灰色地帶→轉介勞工處或工會
```

#### 2.2 修改「核心場景」（最重要的部分）

以「勞動契約」為例的完整改寫：

**修改前**（租約場景1）：
```markdown
### 場景1：對租客不利的條款檢測

決策規則：
- 若合約中出現「不得記載事項」清單中的禁止條款 → 高風險
  - 例：「房東無須返還押金」
```

**修改後**（勞動契約場景1）：
```markdown
### 場景1：違反勞基法強制規定的條款檢測

決策規則：
- **規則 A1**：若合約包含「勞基法第 22 條禁止的薪資扣減」但條款中有扣薪條文 → 高風險
  - 對應條款例：「遲到 1 次扣薪 100 元」、「曠職當月薪資減 50%」
  - 官方規範：中華民國勞基法第 22 條第 2 項「工資應全額直接給付勞工」
  - 合約原文參考：提取合約中「薪資」相關條款，搜尋「扣」「減」「罰」關鍵字

- **規則 A2**：若合約要求試用期間無勞保 → 中風險（違法但常見）
  - 對應條款例：「試用期間不辦理勞保」、「試用期到期後才加保」
  - 官方規範：勞基法施行細則第 2 條「試用期視為勞動契約期間」

- **規則 A3**：若合約包含「禁止員工兼職」無合理限制 → 中風險
  - 合約原文例：「員工不得從事副業」（無「競業禁止」或「時間限制」的正當理由）
  - 法律邊界：法院傾向認為無合理限制的兼職禁令違憲
```

**關鍵修改點**（做改寫時的檢查清單）：

- [ ] 規則來源都對應到具體的法律條文編號
- [ ] 每個規則都附帶「常見違規樣態」的真實例子
- [ ] 每個規則都有清楚的「合約中該搜尋什麼關鍵字」
- [ ] 區分「明確違法」（高風險）vs. 「灰色地帶」（中風險）vs. 「道德問題」（轉介）

#### 2.3 修改「轉介資源」

**修改前**（租約）：
```markdown
崔媽媽基金會（租屋諮詢）：02-xxxx-xxxx
法律扶助基金會：xxx-xxxx
各縣市消保官：1950
```

**修改後**（勞動契約）：
```markdown
勞工處（各縣市）：查詢 https://www.mol.gov.tw/
全國工會聯合會：[聯絡方式]
勞資爭議調解：聯絡當地勞工局申請調解
消費者保護官：1950（如涉及消費爭議）
法律扶助基金會：[聯絡方式]
```

**檢查清單**：
- [ ] 所有轉介電話號碼都是真實的、最新的、能接通的
- [ ] 有機構網址讓使用者自己驗證
- [ ] 如果可能，附上服務時間和接通方式

---

### Step 3：提交給 Claude Code（5 分鐘）

#### 3.1 準備一個測試用例

在提交給 Claude Code 之前，先準備 1–2 份「測試合約」來驗證系統：

```
租約例子→使用真實或匿名化的租約樣本
勞動契約例子→從 PTT 或社群討論中找 3–5 個真實案例
```

#### 3.2 構建提示詞

```prompt
我想構建一個「規則庫比對型 Dynamic Workflow」，自動檢測 [場景] 中的不合規條款。

【定義書】
[複製你修改好的定義書內容]

【測試用例】
[貼入 1–2 份你要測試的合約樣本]

請根據上述定義書，自動生成一個 5-Stage Pipeline（Stage 1：OCR / Stage 2：切分 / Stage 3：規則比對 / Stage 4：解釋生成 / Stage 5：信心評分），能對這份測試合約跑出三段式報告。

【最低要求】
1. 每條風險判斷必須同時引用「合約原文」和「官方規範條目」
2. 低信心判斷（<70%）必須直接轉介，不給結論
3. 永遠不說「可以簽」或「不要簽」，只說「有這些差距，簽前確認」
```

#### 3.3 審查生成的代碼

Claude Code 會輸出一個 Workflow 框架。你需要檢查：

**清單**：
- [ ] Stage 1 OCR 部分是否考慮了照片品質問題（昏暗、斜拍、皺摺）
- [ ] Stage 3 規則比對是否用了規則引擎（而非純 LLM）
- [ ] Stage 4 是否對每條判斷都強制執行「雙引用」（合約原文 + 規範條目）
- [ ] Stage 5 是否有清楚的信心度閾值定義（如 <70% 轉介）
- [ ] 輸出是否包含免責聲明和轉介資源聯絡方式

---

### Step 4：審查生成的代碼（30 分鐘）

#### 4.1 功能驗收清單

對生成的 Workflow 進行以下驗收：

| 驗收項 | 檢查方式 | 預期結果 | 備註 |
|--------|---------|---------|------|
| **雙引用強制機制** | 檢查 Stage 4 的代碼，尋找「if 引用失敗 → 丟棄」的邏輯 | 有明確的代碼判斷 | 這是三大鐵律之一 |
| **信心度評分實現** | 檢查 Stage 5，是否有 0–100 的信心評分計算 | 有清楚的評分公式 | <70% 自動轉介 |
| **轉介話術無結論** | 搜尋輸出中有沒有「可以簽」「不要簽」的字句 | 0 個結論性字句 | 永遠是「確認清楚」而非「決策」 |
| **OCR 降級機制** | 檢查 Stage 1，若 OCR 失敗 >20% 會怎樣 | 輸出「照片看不清」提示 | 不許亂猜 |
| **免責聲明** | 檢查輸出是否在末尾包含免責聲明 | 有清楚的免責文本 | 每次輸出都必有 |

#### 4.2 邏輯驗收清單

```
使用你準備的 1–2 份測試合約，跑過整個 Workflow，問自己：

1. 【漏報檢查】這份合約中的明顯風險，系統有沒有抓到？
   - 如果有漏掉 → 回頭修改定義書（規則是否夠清楚）或回訪 Claude Code

2. 【誤報檢查】系統有沒有把沒問題的東西當成問題？
   - 如果有誤報 → 檢查是否是規則定義太寬鬆

3. 【引用準確性】系統引用的合約原文是否準確，官方規範條目是否真實存在？
   - 如果有錯誤引用 → 回頭修改規則庫定義

4. 【話術檢查】低信心的情況，系統是不是直接轉介而沒有自信結論？
   - 如果有「可以簽」類結論 → 違反鐵律，必須回頭改
```

---

### Step 5：執行和迭代（1 小時 + 持續）

#### 5.1 小規模試運行（1 小時）

在全量評測之前，用 5–10 份合約進行試運行：

```
Day 1：用 5 份合約跑 Workflow
  ↓
檢查結果（3 大問題類）：
  1. 系統崩潰或無法完成（技術問題）
  2. 系統輸出但品質堪憂（邏輯問題）
  3. 系統正常但用戶不信任（信任問題）

Day 2：根據問題修改
  - 技術問題 → 改 Workflow 代碼
  - 邏輯問題 → 改定義書
  - 信任問題 → 改話術與轉介資源

Day 3：再跑 5 份，驗證修改
```

#### 5.2 建立「評測標準」文檔

根據你的場景，建立評測標準：

**租約哨兵的例子**：
```markdown
# 評測標準 v1.0（勞動契約版）

## Ground Truth 標注規則

每份合約要標注：

### 「違反勞基法」級陷阱（高風險）
- 例1：要求試用期不上保險
- 例2：薪資中扣除法定之外的項目
- 例3：強制加班無加班費

### 「保護不足」級陷阱（中風險）
- 例1：未明確列出試用期天數
- 例2：未說明加班費計算方式
- 例3：未提離職金

### 「灰色地帶」（轉介）
- 例1：競業禁止條款的合理性判斷
- 例2：特定職位是否屬「監督管理人員」（豁免加班）
```

#### 5.3 完整評測流程（第 2 週開始）

```
週 1：試運行（5–10 份）→ 修改
週 2：小規模評測（15 份）→ 計算漏報/誤報率
週 3：完整評測（25 份 × 2 輪）→ 發布機房報告
週 4：真人測試（5 位使用者）→ 決策是否上線
```

## 進階技巧

### 技巧1：為不同場景自定義決策規則

如果你要同時支援「房東版本」（房東想檢查自己的合約是否會被告）和「租客版本」（租客想檢查是否被坑），怎麼辦？

**做法**：在定義書中添加「場景切分」：

```markdown
## 多角度規則庫

### 租客角度（預設）
- 「這份合約對我不利嗎？」
- 檢查標準：「對租客有利」的官方規範

### 房東角度（進階）
- 「這份合約會讓我承擔過多風險嗎？」
- 檢查標準：「對房東合理」的法律界線（不同於「對租客有利」）

【提示詞】
請根據「使用者角度參數」，生成不同的決策規則集。
使用者傳入 `角度: "租客"` 時→執行租客版規則
使用者傳入 `角度: "房東"` 時→執行房東版規則
```

### 技巧2：為組織定制化「轉介資源」

不是每個組織都想轉介給政府機構。有些可能想轉介給自己的法律部門、工會、或 NGO。

```markdown
## 轉介資源配置（可定制）

在 Workflow Stage 5，可以注入一份 JSON 配置：

{
  "default_referral": {
    "tenant_side": "台灣租人聯盟",
    "landlord_side": "地主協會",
    "government": "內政部"
  },
  "org_specific": {
    "大學": "學生會法律委員會",
    "工廠": "工會代理人",
    "小商家": "中華民國全國商業會"
  }
}
```

### 技巧3：添加「優先級過濾」

如果合約有 100+ 個問題，系統應該優先報告什麼？

```markdown
## 優先級過濾（可選進階功能）

Stage 5 補充：若風險數量 > 5 條，按以下優先級排序：

1. 【致命】直接違反法律禁止項 → 必報
2. 【高風險】常見的坑點 → 必報
3. 【中風險】保護不足但不違法 → 報
4. 【低風險】細節瑕疵 → 按使用者設定「詳細模式」決定是否報

[提示詞]
請在 Stage 5 加入優先級排序邏輯，並允許使用者選擇「快速版」（只報致命+高風險）或「詳細版」（全部報）。
```

## 案例演練

### 案例1：台灣租約 → 應用到香港租約

**背景**：香港的租屋市場同樣有弱勢租客被坑的問題，但法律框架（《房屋條例》）與台灣不同。

**修改過程**：

**Step 1**：獲取香港官方規範
```
官方來源：香港特區政府地政總署《住宅物業管理手冊》+ 《房屋條例》第 46 條（不公平條款）
```

**Step 2**：改寫決策規則
```markdown
【修改前】（台灣）
- 若出現「房東無須返還押金」→ 違反內政部規範

【修改後】（香港）
- 若出現「業主可任意扣除按金」且無清晰退回條件 → 違反《房屋條例》第 46 條（不公平條款）
- 若租期 <2 年但按金 > 2 個月租金 → 違反香港慣例（非法律禁止，但風險高）
```

**Step 3**：改寫轉介資源
```markdown
【修改前】（台灣）
- 崔媽媽基金會
- 法律扶助基金會

【修改後】（香港）
- 香港撮合及爭議解決中心（中介糾紛調解）
- 香港律師會法律援助委員會（法律援助）
- 房屋署投訴部門（政府住房糾紛）
```

**成果**：變成「香港租約檢測 Workflow」，無需改一行代碼，只改定義書。

### 案例2：租約檢測 → 應用到消費者合約

**背景**：許多消費者在購買電子產品、簽署會員服務時被「霸王條款」坑。

**改寫思路**：

```markdown
【原始場景】租約
- 檢查對象：住房租賃合約
- 官方規範來源：內政部住宅租賃規範

【新場景】消費合約
- 檢查對象：消費者合約（購買商品、訂閱服務、會員條款等）
- 官方規範來源：消費者保護法 + 各行業標準（如電信消費契約範本）
```

**核心改動**：

| 項目 | 租約 | 消費合約 |
|------|------|---------|
| **不得記載事項** | 「房東無須返還押金」 | 「消費者無法退貨」、「商家無責任承諾品質」 |
| **應記載事項** | 「租期」、「押金返還條件」 | 「退貨期限」、「保修期限」、「客服聯絡方式」 |
| **嚴重度分級** | 涉及經濟損失的陷阱 | 涉及隱私（蒐集個資不同意）或無法撤銷（自動續訂） |
| **轉介資源** | 房東糾紛調解機構 | 消費者保護官、行業協會調解 |

**成果**：同一個 Workflow 框架，改定義書就能服務「租約 / 勞動契約 / 消費合約 / 保險契約」等。

---

### 案例3：用 Claude Code 迭代修復

**情景**：W2 評測時發現系統漏掉了「定型化條款可否加重消費者責任」這一類陷阱。

**修復流程**：

1. **診斷**：定義書中「應記載事項」遺漏了這類風險
2. **回頭改定義書**：
   ```markdown
   新增規則 B4：
   - 若合約要求消費者承擔「超出合理範圍」的責任 → 中風險
     例：購買日用品卻要簽「若用壞了須賠 100 倍價格」
     對應法律：消費者保護法第 11–12 條（不公平條款禁止）
   ```
3. **提交修改給 Claude Code**：
   ```prompt
   定義書已更新，請在 Stage 3 規則庫中補充規則 B4 的比對邏輯。

   規則 B4 需要檢查的合約關鍵字：「賠償」、「責任」、「損害」
   規則 B4 的信心度評分邏輯：
   - 若責任金額明確超出常理（如日用品賠 100 倍）→ 高信心（95%）
   - 若責任金額不清楚需要專業判斷 → 低信心（40%）→ 轉介
   ```
4. **重新評測**：用 25 份合約跑第 2 輪，檢查漏報率是否下降
5. **文檔記錄**：在評測報告中標注「漏報 #5（規則 B4）：修復完成」

## 常見問題

### Q1：我不是開發者，可以跑這個流程嗎？

**A**：完全可以。關鍵步驟是定義書編寫和評測，都不涉及編碼。你需要：
- ✅ 能讀懂你國家的法律文件
- ✅ 能找到 3–5 份真實合約做測試
- ✅ 能驗證系統的輸出是否正確（用人工判斷）
- ❌ 不需要會寫代碼

若要運行系統，也可以請開發者幫你部署 Claude Code，你只需提供定義書即可。

---

### Q2：定義書寫好了，生成的 Workflow 品質怎麼保證？

**A**：三層保證：

1. **代碼層**：Claude Code 自動加入「雙引用強制」「信心度閾值」「轉介機制」等護欄
2. **評測層**：你必須用 25+ 份合約進行標注和驗證，計算漏報/誤報率
3. **人類層**：W4 真人測試，請真實用戶評估「你敢不敢信任這個系統」

如果任何一層沒通過，系統不上線。

---

### Q3：如果我的國家沒有「官方規範文件」，怎麼辦？

**A**：這時「規則庫比對型 Workflow」這個模式本身就不適用。改用「風險提示型」Workflow：

```
不是：檢查「是否符合官方規範」
而是：給出「常見陷阱提示」（例如「大多數租約不會主動寫明押金退還期限」）
```

但這種模式的信心度會降低，更容易誤導使用者，所以：
- ❌ 不推薦用於「高風險決策」（如租約、勞動契約）
- ✅ 可用於「低風險提醒」（如產品評測、服務建議）

---

### Q4：生成的代碼有 bug，怎麼辦？

**A**：Bug 修復流程：

1. **描述 bug**：「Stage 2 對某些格式的合約無法正確切分」
2. **提供測試用例**：貼出出現 bug 的合約樣本 + 期望輸出 + 實際輸出
3. **提交修復請求**：
   ```prompt
   這份合約在 Stage 2 出錯（細節見下）。

   【出錯的合約】
   [貼入合約內容]

   【期望輸出】
   [期望的切分結果]

   【實際輸出】
   [系統給出的錯誤結果]

   請修復 Stage 2 的切分邏輯。
   ```
4. **迭代驗證**：修復後用原測試用例驗證

---

### Q5：系統上線後發現有漏報，怎麼辦？

**A**：這是常見且可以接受的情況。應對流程：

1. **記錄漏報**：「這份合約有 X 陷阱，系統沒抓到」
2. **分類漏報**：
   - **定義書缺陷**：規則本身沒定義這類陷阱 → 修改定義書
   - **實現缺陷**：規則有定義但 Workflow 沒執行好 → 改代碼
   - **邊界情況**：太灰色，無法自動判斷 → 正常轉介
3. **改進定義書或代碼**，進行後續版本更新（v1.1、v1.2）
4. **持續監控**：每季或每年回顧一次真實漏報數據，推動迭代

---

### Q6：能為多個國家同時生成嗎？

**A**：理論上可以，但需要小心設計：

```markdown
## 多國家配置方案

Workflow 接受一個 `region` 參數：

if region == "TW":
    加載台灣定義書 → 執行台灣規則庫
elif region == "HK":
    加載香港定義書 → 執行香港規則庫
elif region == "JP":
    加載日本定義書 → 執行日本規則庫
```

**但警告**：
- 每個國家都要做完整評測（不能共享評測數據）
- 轉介資源必須是「各國的真實組織」，不能馬虎
- 免責聲明要符合各國法律

建議：先把一個國家做到可靠上線，再逐步加入其他國家。

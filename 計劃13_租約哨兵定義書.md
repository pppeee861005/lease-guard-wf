# 計劃13：租約哨兵定義書

**狀態**：🔄 規劃中
**版本**：v1.0（與30天專案計劃配套）
**用途**：定義業務邏輯，供 Claude Code 自動生成檢測 Workflow

---

## 一、問題定義

### 1.1 問題陳述

**清晰的問題敘述**：

自動評估使用者上傳的租約照片（可能昏暗、斜拍、皺摺），與內政部「住宅租賃定型化契約應記載及不得記載事項」逐條比對，輸出三類問題：
1. ⚠️ 對租客不利的條款（並附合約原文截句與官方規範條目）
2. ✅ 該有但沒有的保護條款
3. 📍 無法自動判斷時的轉介路徑（崔媽媽基金會/法律扶助基金會/消保官）

### 1.2 背景與約束

**背景**：
- 台灣租屋市場中，弱勢租客（學生、低收入戶）常因資訊不對稱而被坑
- 一次被坑的押金或惡意條款可能造成數月甚至數年的經濟損失
- 現有免費法律諮詢資源有限，難以應對高頻詢問
- AI 技術使「每人一個顧問」成為可能，但需要極高的精準度以避免二次傷害

**約束條件**：
- 不構成法律意見（明確免責聲明）
- 寧可漏報轉介，不可錯報安心（三大鐵律之一）
- 永遠保留人類介入的出口（低信心 → 轉介）
- 設計時內建「閱後即焚」（不留存使用者合約）

### 1.3 輸入與輸出

**輸入（Input）**：
- **類型**：手機拍攝或掃描的租約照片
- **規模**：單份租約通常 3–15 頁
- **格式**：JPG/PNG，分頁上傳；照片品質涵蓋掃描件、正拍、斜拍昏暗
- **前置條件**：使用者已同意使用條款與閱後即焚原則

**輸出（Output）**：
- **類型**：結構化的三段式報告
- **格式**：純文字（無 HTML）+ 聯絡方式（電話/網址）
- **必須包含**：
  - 每條風險判斷的合約原文精確字句
  - 每條風險判斷對應的官方規範條目編號
  - 信心度評分（若低信心則直接轉介）
  - 免責聲明與人類資源聯絡方式

## 二、核心場景

### 場景1：對租客不利的條款檢測

**背景**：
系統收到租約照片後進行 OCR，將文本結構化，逐條與官方「不得記載事項」清單比對，並檢查「應記載事項」中的必要保護條款是否被修改成對租客不利的版本。

**決策規則**：
- **規則 A**：若合約中出現「不得記載事項」清單中的禁止條款 → 判定為「違反官方規範」→ **輸出警示（高風險）**
  - 例：「房東無須返還押金」、「租客不得請人維修」、「租客應承擔房屋稅」等
  - 附帶：合約原文字句（20–30 字截句）+ 官方規範條目編號 + 嚴重度等級

- **規則 B**：若「應記載事項」中的條款被大幅修改、削減或移除 → 判定為「保護不足」→ **輸出警示（中風險）**
  - 例：官方規範要求「明確列示租賃期限」，合約卻只寫「月租」，沒有起迄日期
  - 修復建議：「這部份應該要寫 [具體內容]，建議簽名前先問房東」

- **規則 C**：若對照模糊或需要法律解釋 → **轉交信心評分模組 → 根據信心值決定是否轉介**

**成功指標**：
| 指標 | 衡量方式 | 目標值 |
|------|---------|--------|
| 高嚴重度漏報率 | 25+份測試合約，高等級陷阱漏掉的比例 | <5% |
| 誤報率 | 真正沒問題卻被標記為有問題的比例 | <3% |
| 引用準確率 | 系統引用的合約原文是否準確對應 | >95% |

---

### 場景2：該有但沒有的條款檢測

**背景**：
房東可能刻意遺漏官方要求「應記載」的保護條款，以規避法律責任或隱藏不公平條款。系統需要檢測這些缺失。

**決策規則**：
- **規則 D**：比對官方「應記載事項」清單，若合約中完全缺失某些必要條款 → **輸出提醒（中風險）**
  - 例：缺少「押金返還條件與期限」、「雙方聯絡方式」、「維修責任劃分」等
  - 話術：「這份合約沒有寫 [條款名]，但法規要求應該要有。建議簽名前問清楚房東的立場。」

- **規則 E**：若所有應記載項都齊全 → **輸出綠燈**「該有的都有了，恭喜」

**成功指標**：
| 指標 | 衡量方式 | 目標值 |
|------|---------|--------|
| 缺失偵測準確率 | 比對 ground truth 標注 | >90% |
| 誤判率 | 把「寫在其他地方」當作「缺失」的比例 | <5% |

---

### 場景3：無法自動判斷 → 轉介決策

**背景**：
某些條款涉及灰色地帶、法律邊界不清、或需要結合當地法規的複雜判斷，系統應自動轉介給人類專業人士而非亂給建議。

**決策規則**：
- **規則 F**：若系統對某條判斷的信心度 <70% → **強制轉介**
  - 標準話術：「這一條我看不太準，建議你帶著合約洽詢：
    - 崔媽媽基金會（租屋諮詢）：02-xxxx-xxxx
    - 法律扶助基金會：xxx-xxxx
    - 你所在縣市消費者服務中心：1950」
  - **不允許系統說「可以簽」或「不要簽」**（永遠是分流，不是決策）

- **規則 G**：若 OCR 失敗率 >20%（照片太糊看不清）→ **輸出「照片看不清，請重拍這幾頁」**

**成功指標**：
| 指標 | 衡量方式 | 目標值 |
|------|---------|--------|
| 轉介適當率 | 真正應該轉介的案件是否都被轉介了 | >95% |
| 過度轉介率 | 本可以判斷卻轉介了的比例 | <10% |

## 三、驗證標準

### 3.1 成功指標（總體）

| 指標 | 衡量方式 | 目標 |
|------|---------|------|
| **漏報率（高嚴重度）** | 25+份測試合約，高等級風險被系統漏掉的比例 | <5% |
| **誤報率** | 沒問題的條款被誤判為有問題的比例 | <3% |
| **引用準確率** | 系統輸出的合約原文截句是否準確對應 | >95% |
| **OCR 可用率** | 不同照片品質下 OCR 成功率 | 正常照：>90%；劣質照：>70% |
| **轉介準確率** | 該轉介的被轉介，不該轉介的被自動判斷 | >95% |

### 3.2 驗收條件

- [ ] 核心 pipeline（5 個 Stage）端到端可跑
- [ ] 25+份合約完整評測完成（包含漏報覆盤）
- [ ] 5 位真人使用者測試完成，信任度反饋記錄
- [ ] 所有輸出都包含雙引用（合約原文+規範條目）
- [ ] 低信心判斷全部轉介，無例外
- [ ] 免責聲明與人類資源聯絡方式在每份報告中清晰可見

### 3.3 失敗標準

- 若高嚴重度陷阱漏報率 >10% → 暫停啟動 LINE bot，轉為修復週
- 若系統輸出「可以簽」或「不要簽」類結論性語句 → 違反鐵律，回頭重寫
- 若合約原文引用不準確（引用準確率 <90%）→ 禁止上線
- 若轉介決策的信心閾值沒有代碼層落地 → 回頭補代碼

## 四、執行策略

### 4.1 決策規則（規則引擎層）

```
輸入合約照片 → OCR 與前處理
  ↓
遍歷官方「不得記載事項」清單
  若找到匹配 → 規則A 判斷（高風險）

遍歷官方「應記載事項」清單
  若條款被修改 → 規則B 判斷（中風險）
  若條款缺失 → 規則D 判斷（中風險）

對每條判斷進行信心評分
  若信心 <70% → 規則F：強制轉介
  若信心 ≥70% → 雙引用強制檢查
    引用成功 → 輸出警示
    引用失敗 → 該判斷被丟棄（寧漏勿誤）

輸出三段式報告
```

### 4.2 降級條件（保護機制）

- **Token 消耗達 80%** → 停止低優先級檢查（某些灰色地帶判斷），標記為 PARTIAL_COMPLETION，明確告知使用者
- **OCR 失敗率 >20%** → 降級為「照片看不清」提示，不嘗試用 LLM 補救
- **規則庫版本過時** → 自動標注規則庫抓取日期，每季與官方來源比對

### 4.3 升級條件（人類介入）

- **系統無法判斷（信心 <70%）** → 升級轉介，提供崔媽媽/法扶聯絡方式
- **合約有特殊條款（如涉及保證人、居家工作等）** → 升級轉介
- **使用者有後續爭議** → 建議走正式調解或訴訟管道

## 五、預算分配

### 5.1 預算概況

**總預算**：預估 50 萬 tokens（相當於約 15–20 次完整的 Sonnet 評測）
**預留率**：≥50%
**說明**：
- W2 核心 pipeline 開發：15–20 萬 tokens（OCR 最耗費）
- W3 完整評測（25 份合約）：15–20 萬 tokens
- W4 真人測試與改進：10 萬 tokens
- 預留 30%：應急調整與修復

### 5.2 成本估算

| 項目 | 成本(估計) |
|------|-----------|
| Stage 1 OCR + 影像前處理 | 120,000–150,000 tokens |
| Stage 2 條款切分與結構化（25份×Haiku） | 30,000–50,000 tokens |
| Stage 3–4 規則比對 + 生成（25份×Sonnet） | 80,000–120,000 tokens |
| Stage 5 信心評分 + 轉介決策 | 20,000 tokens |
| 完整評測循環（第1輪+第2輪） | 50,000 tokens |
| 修復與迭代 | 30,000 tokens |
| **總計** | **約 300,000–500,000 tokens** |

### 5.3 監控機制

- **關鍵監控點**：50%、80%、95%
- 每日記錄實際消耗 token 數（不估不預）
- 若達到 80% → 評估是否進入砍範圍或降級執行模式
- 成本超支應急方案：砍 W4 真人測試的份數（從 5 降為 3），但保留核心評測

## 六、DW 應用優勢

### 與傳統開發相比

```
傳統律師方式：
  只有有錢的人能承擔 → 租客無保護

傳統代碼定義：
  開發者自行設計邏輯 → 每次修改都改代碼 → 很難被非技術人員理解

✨ DW + 規則庫方式：
  定義 = 官方規範（來自內政部，公開透明）
  邏輯 = 規則引擎（可被非技術人員讀懂、驗證、修改）
  執行 = Claude Code 自動生成 + LLM 補充（快速迭代）

結果：
  - 每個組織可以複製這份定義書，換成自己國家的官方規範，就能用
  - 法律邊界模糊時改定義書而非改代碼
  - 任何人都能驗證邏輯是否公平
```

## 七、社群複用性（Fork 指南）

本定義書的結構可應用於其他國家的租賃法或其他「政府規範已定，但沒人替弱勢者執行」的場景：

**需要替換的**：
1. `規則庫` → 你所在國家/地區的租賃法或相關法規
2. `轉介資源` → 當地真實存在、真的接得住人的組織
3. `免責聲明` → 符合當地法律的免責文本

**不能改的**：
- 三大鐵律（寧漏勿誤 + 轉介人類 + 分流不決策）
- 雙引用強制機制（引用出不來就丟棄）
- 評測優先於上線的紀律

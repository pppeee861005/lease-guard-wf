可以設計,但不該用在你想的位置。讓我把這個問題拆成兩半:「用 agent 去查內政部網頁」是好主意;「不建立資料庫」是壞主意。正確答案是兩者結合——快照 + 哨兵架構。

先說為什麼「執行時即時查詢、不存規則庫」這條路走不得,有四個否決理由,每一個都直接撞上你自己定的鐵律:

第一,可審計性會死。 你的機房報告要寫「本系統在規則庫 v2026-06-15 版上,對 25 份合約的漏報率為 X%」。如果規則是執行時即時抓取、即時解析的,同一份合約今天和下週跑出的結果可能不同——評測不可重現,收據就不成立。信任生產線的地基是「凍結的、有版本號的事實」,即時抓取在地基上挖了個洞。

第二,鐵律一會被架空。 「每條判斷強制雙引用規範條目」的前提是規範以結構化形態存在(條目編號、原文、嚴重度)。即時抓取意味著每次用戶上傳合約,LLM 都要現場解析一次政府網頁的 HTML,再從解析結果裡引用——每一次執行都新增一次解析出錯或幻覺的機會。規則庫是把「理解官方規範」這件高風險工作做一次、人工驗證一次、然後凍結;即時查詢是把它重做一萬次、每次都沒人驗證。對一個服務脆弱用戶的系統,這是把最該確定的東西變成了最不確定的。

第三,經濟學不通。 法規幾年改一次,合約每天進來。為一個年變動率趨近於零的數據源,付每次請求的抓取與解析成本(token、延遲、政府網站偶爾掛掉或改版),是把快取該解決的問題交給了最貴的工具。用戶在昏暗房間裡等報告,你不會希望瓶頸是政府網站當天的響應速度。

第四,fork 資產會消失。 README 裡寫了 rules/ 是本 repo 最有價值的資產之一——其他國家的人移植時,換掉的就是這個目錄。如果規則活在執行時的網頁抓取邏輯裡,這個專案就沒有可以被 fork 的核心,只剩一段綁死台灣某個網址的爬蟲。

那 agent 該放在哪?放在規則庫的保鮮層,這才是你直覺裡正確的那一半:

[規則哨兵 Agent](排程:每週或每月一次)
   ↓
抓取內政部官方頁面 → 與 rules/ 現行快照做 diff
   ↓
無變動 → 記錄「YYYY-MM-DD 核對,無更新」(這行紀錄本身就是審計軌跡)
有變動 → 自動開 GitHub Issue/PR,附上條文 diff
   ↓
人工審核變動 → 合併 → 規則庫版本號 +1 → 機房戰報記錄一筆
注意最後一步是人工審核,不是自動合併——規則庫是系統的憲法,憲法修正案不能由 agent 自己批准。這個設計同時解決了你擔心的「資料庫過時」問題(風險登記簿裡的那條),而且把維護成本壓到每月幾分鐘。

至於 W1 那 2–3 小時的結構化苦工,agent 同樣可以代勞,但分工要清楚:agent 起草,人類逐條驗收。流程是你下載官方文件 → agent 把每一條轉成結構化 JSON(條目編號、原文、白話解釋、常見違規樣態、嚴重度初判)→ 你逐條核對原文是否一字不差、嚴重度是否合理。原文抄錄的部分 agent 幾乎不會錯,但「嚴重度分級」和「常見違規樣態」涉及判斷,這部分的最終簽名必須是你的——還是那條原理:AI 沒有法律責任,所以憲法級的內容它只能起草,不能定稿。

最後一個彩蛋:這個「快照 + 哨兵」模式本身就是一篇教科書文章的素材。它是「規則庫比對型 WF」的標準配件——任何規則庫(法規、補助清單、API 文檔)都面臨同樣的保鮮問題,而答案永遠是這個三件組:凍結的版本化快照、定期 diff 的哨兵 agent、人工審批的修憲流程。等租約哨兵和未來第二個顧問席都用上它,兩次實證達標,就夠資格進教科書層了。

一句話總結:執行管線永遠不碰網路,規則庫永遠有版本號,哨兵負責讓快照不過期,人類負責簽名。 你的直覺(讓 agent 去查網頁)是對的,只是它的崗位在後勤,不在前線。

## 🔄 日誌系統的運作規則

### 日常（工作日）

**每日工作結束前**（5–10 分鐘）：
1. 更新進度追蹤表（打勾完成的任務、記錄當日燃燒的 tokens）
2. 簡短備忘（遇到的坑、明天優先做什麼）

### 周末（覆盤日）

**每週六**（30 分鐘）：
1. 將周日誌匯總成 `work_log_YYYYMMDD.md`
2. 填寫周覆盤（本週最大的坑、規則庫中最難的條目、偏離計劃之處）
3. 更新日誌索引中的「版本 v1.0」表格（進度更新）
4. 更新指標儀表板

**周日**：神聖休息日，不開任何工作表

### 里程碑完成時

**每當完成一個里程碑**（如 W2 結束時）：
1. 發布對應的報告（如 W3 的機房報告 #1）
2. 在日誌中記錄「是否達到見真人的門檻」
3. 更新進度追蹤表的「覆盤」部分

## 🎯 成功的定義（30 天版）

本計劃成功的唯一標準：

1. ✅ **一條可重現的 pipeline**
   - 任何人 clone repo 後能對一份合約跑出三段式報告

2. ✅ **一份公開的、含難看數字的評測報告**

3. ✅ **5 位真人的真實反饋**
   - 哪怕結論是「還不能用」，也比一個人悶著開發強

4. ✅ **「規則庫比對型 Workflow」作為可 fork 的模式被文檔化**
   - 教學指南 + 案例 + fork 指南完整

**不在成功定義裡的**：
- ❌ 用戶數、訂閱數、任何成長指標
- ❌ 商業化或融資
